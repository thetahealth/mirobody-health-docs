Mirobody 为全球与中国区工作负载设置独立的区域环境。全球生产集群目前可用；中国区生产 API 与公开 `.cn` 端点仍在上线过程中。选择 base URL 前，请先阅读[区域](/zh/api-reference/regions/overview)。

本页说明 API 已提供的产品控制，不构成笼统的合规认证声明。具体监管义务取决于使用场景、合同、数据流与所在区域。

## 产品控制

### Subject 隔离

每个经过认证的 `/v1` 请求都限定在 API key 所属账户内。在账户内部，`user` 字段映射到隔离的 Subject。开发者无法使用自己的 API key 读取其他账户的 Subject。

建议为 `user` 使用稳定且不敏感的标识符。除非你的数据策略明确要求，否则不要在该字段中放入邮箱、病历号或其他直接身份标识。

### API 凭据

API key 使用 `mb_live_*` 格式。完整密钥只在创建时返回一次；服务端保存其哈希，开发者控制台之后仅显示 key 前缀。如怀疑密钥泄露，请立即撤销。

### 留存与删除

API 为各类存储面提供显式控制：

| 范围 | 控制 |
| --- | --- |
| 结构化记录 | `DELETE /v1/data`；限时留存（`1h`、`2h`、`6h`、`1d`） |
| 文件 | `DELETE /v1/files/{file_key}`；限时或会话留存 |
| 存储响应 | `DELETE /v1/responses/{id}` |
| 会话范围的数据与文件 | `DELETE /v1/sessions/{id}` |
| 整个 Subject | [`DELETE /v1/subjects/{user}`](/zh/api-reference/lifecycle#下线一个-subject) |

`DELETE /v1/sessions/{id}` **不会**删除已存储的 Responses API 对象。各删除路径的准确行为见[数据生命周期](/zh/api-reference/lifecycle)与[状态与记忆](/zh/api-reference/state-and-memory)。

### 可追溯输出

Agent 响应可返回 `tool_steps`、`health_records` 与 `citations`，便于应用检查回答所使用的工具调用与证据。这些字段是执行轨迹，不代表会暴露模型的私有推理或所有内部操作。

在 Agent API 上，`store:false` 会阻止响应对象与对话线程在请求结束后保留，也会使该轮对话不进入异步记忆固化。

### 区域处理

不同区域环境使用独立主机与存储。除非你的合规审查允许由此产生的数据传输，否则不要为同一 Subject 切换区域。集群可用性与数据驻留要求应在接入时确认。

## 法规审查

中国区部署可能适用包括以下法律在内的要求：

| 法律 | 官方文本 |
| --- | --- |
| 《个人信息保护法》 | [全国人民代表大会](http://www.npc.gov.cn/npc/c2/c30834/202108/t20210820_313088.html) |
| 《数据安全法》 | [国务院](https://www.gov.cn/xinwen/2021-06/11/content_5616919.htm) |
| 《网络安全法》 | [国务院](https://www.gov.cn/xinwen/2016-11/07/content_5129723.htm) |

面向美国医疗场景时，请评估 [HIPAA](https://www.hhs.gov/hipaa/index.html) 义务，并确认是否需要业务伙伴协议（BAA）。不要仅依据 API 能力推断某项工作负载符合 HIPAA。

## 安全与合规资料

如需安全问卷、架构审查、数据处理条款，或确认当前是否可提供 DPA、BAA 与独立评估材料，请联系 [Mirobody Support](mailto:developer@thetahealth.ai)。在发送受监管的生产数据前，应先取得适用文件。
