**Mirobody API** 把杂乱的健康数据变成**标准化、AI 可用的记录** —— 并提供两个兼容 OpenAI 的接口让 agent 在其上工作。它是[开源 Mirobody 引擎](/zh/welcome)之上的托管层：我们托管存储、模型与密钥；你只需把 OpenAI SDK 指向 base URL 并带上 `mb_live_*` 密钥。

## 闭环：上传 → 标准化 → AI 使用

<Steps>
  <Step title="按数据形态选择入口">
    结构化读数走 [`POST /v1/data`](/zh/api-reference/data)。[`POST /v1/files`](/zh/api-reference/files) 保存化验 PDF、照片与表格，并抽取其文本；需要把报告转成标准化读数时，使用 [`POST /v1/extract`](/zh/api-reference/extract)。
  </Step>
  <Step title="每条结构化读数都会标准化">
    名称确定性解析到 **LOINC**（不让 LLM 猜码），值归一到 **UCUM** 单位，每条读数都有 **FHIR** 镜像。`"血糖(空腹)"`、`"FBG"`、`"Glucose, fasting"` 归成一条序列。见[标准化](/zh/api-reference/standardization)。
  </Step>
  <Step title="让 AI 在其上工作">
    向 [Answers API](/zh/api-reference/chat) 要一条有依据、带证据的回答 —— 或在 [Agent API](/zh/api-reference/responses) 上构建完整 agent：你自己的工具、存储的对话，openai-agents SDK 开箱即用。
  </Step>
</Steps>

## 为什么选 Mirobody

- **有依据，而非听着合理** —— 回答可附 `health_records`、`citations` 与服务端采集证据时使用的工具步骤。
- **结构化数据标准化** —— 通过 `/v1/data` 写入或由 `/v1/extract` 存储的记录都会经过 LOINC + UCUM + FHIR；你的分析与智能体读取同一份序列。
- **双重 OpenAI 兼容** —— Chat Completions *与* Responses 两个协议；现有 SDK 与 agent 框架直接可用。
- **天生多租户** —— 一个 key，每个终端用户一个隔离 Subject；按 Subject 的[被遗忘权删除](/zh/api-reference/overview#端点一览)。
- **内核开放** —— 同一引擎[开源](/zh/welcome)；随时可以自托管。

## 运行 Mirobody 的两种方式

- **托管 API**（本平台）—— 用 `mb_live_*` 密钥对接我们的托管集群；存储、模型与密钥由我们代运维。从[快速开始](/zh/api-reference/quickstart)入手。
- **开源** —— `git clone` [Mirobody 引擎](/zh/welcome)，用 `./build.sh` 构建后自己运行。
