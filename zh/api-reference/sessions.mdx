title: "会话"

---

## 端点

```http
DELETE /v1/sessions/{session_id}
Authorization: Bearer mb_live_*
```

将会话标记为已结束。**注意：** 在当前集群上，该调用返回 `deleted: 0`，且**不会**清除通过 [`POST /v1/data`](/zh/api-reference/data) 以 `retention=session` 写入的记录 —— 那些记录仍会在 24 小时会话上限时过期。不要把它当作你唯一的被遗忘权机制。

```bash
curl -X DELETE "https://test-mirobody-api.thetahealth.ai/v1/sessions/sess_abc123?user=alice" \
  -H "Authorization: Bearer $MIROBODY_API_KEY"
```

<Note>
删除会话时请带上写入该会话数据时所用的**同一个** `user`；省略则指向账号的默认 Subject。
</Note>

响应：

```json
{ "status": "ok", "session_id": "sess_abc123", "deleted": 0, "subject": "alice" }
```

| 字段 | 说明 |
| --- | --- |
| `status` | 成功时为 `"ok"`。 |
| `session_id` | 被结束的会话 id。 |
| `deleted` | 被清除的记录条数。对于通过 `POST /v1/data` 写入的记录，**目前返回 `0`** —— 见上方注意事项。 |
| `subject` | 该会话所属的数据主体(Subject)。 |

## 会话如何工作

在 [`/v1/chat/completions`](/zh/api-reference/chat) 中，`session_id` 有两个作用：

1. **串联多轮上下文**——在共享同一 `session_id` 的多次调用之间贯穿上下文。
2. **限定 `retention=session` 数据的范围**——以 `retention=session` 写入的记录 / 文件 / 内联 `health_context`，其存续时间仅与会话相同。

无论如何，会话数据的上限为 **24 小时**。（`DELETE /v1/sessions/{id}` 旨在更早将其清除，但目前对 `POST /v1/data` 写入的记录报告 `deleted: 0` —— 请依赖 24 小时上限。）

```python Python
client.chat.completions.create(
    model="mirobody-flash",
    messages=[{"role": "user", "content": "Here are today's readings — anything off?"}],
    user="alice",
    extra_body={
        "session_id": "sess_abc123",
        "retention": "session",
        "health_context": [{"indicator": "systolic_bp", "value": 148, "unit": "mmHg", "time": "2026-06-16T08:00:00Z"}],
    },
)
# ... later, when the conversation ends:
# DELETE /v1/sessions/sess_abc123  → marks the session ended (see caveat)
```

完整模型参见 [留存](/zh/api-reference/overview#数据留存)。
