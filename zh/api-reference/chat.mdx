## 端点

```http
POST /v1/chat/completions
Authorization: Bearer mb_live_*
Content-Type:  application/json
```

兼容 OpenAI。智能体用内置工具采集数据主体(Subject)的真实健康数据，再组织出**一段干净的回答**，并附带可追溯的引用。设 `stream: true` 以使用 SSE。

## 快速开始

<CodeGroup>
```bash curl
curl https://test-mirobody-api.thetahealth.ai/v1/chat/completions \
  -H "Authorization: Bearer $MIROBODY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
        "model": "mirobody-flash",
        "messages": [{"role": "user", "content": "How is my fasting glucose trending?"}],
        "user": "alice"
      }'
```

```python Python (OpenAI SDK)
from openai import OpenAI

client = OpenAI(api_key="$MIROBODY_API_KEY", base_url="https://test-mirobody-api.thetahealth.ai/v1")

resp = client.chat.completions.create(
    model="mirobody-flash",
    messages=[{"role": "user", "content": "How is my fasting glucose trending?"}],
    user="alice",                       # tenant-isolation key (a Subject)
    extra_body={"include_tool_steps": True},    # Mirobody extensions go in extra_body
)
print(resp.choices[0].message.content)
# resp.choices[0].message.reasoning_content     # deep-thinking (CoT) detail (may be empty)
# resp.choices[0].message.tool_steps            # tool-call trace: {id, name, arguments, result}
```

```javascript Node.js (OpenAI SDK)
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.MIROBODY_API_KEY,
  baseURL: "https://test-mirobody-api.thetahealth.ai/v1",
});

const resp = await client.chat.completions.create({
  model: "mirobody-flash",
  messages: [{ role: "user", content: "How is my fasting glucose trending?" }],

  include_tool_steps: true,
});
console.log(resp.choices[0].message.content);
```

## 请求体

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `model` | string | `mirobody-flash`（默认）或 `mirobody-expert`。详见[模型](/zh/api-reference/models)。 |
| `messages` | array | OpenAI `{role, content}`。`system` 消息会被折叠进最后一轮用户消息，作为调用方指令（详见[系统提示词](#系统提示词)）。 |
| `stream` | bool | `true` → `chat.completion.chunk` 帧的 SSE 流。 |
| `user` | string | 多租户隔离键 → 一个数据主体(Subject)。传入每个终端用户的稳定 id。详见[多租户](/zh/api-reference/overview#多租户：user-字段)。 |
| `retention` | string | `permanent`（默认）/ `1h` / `2h` / `6h` / `1d` / `session` / `none` —— 应用于你附带的任何 `health_context`。小时级粒度表示写入后约这么多小时自动删除（上限仍硬性限制为 ≤ 24 小时）。 |
| `session_id` | string | 当 `retention=session` 时必填；同时用于串联多轮上下文。 |
| `health_context` | array | 本次请求的内联健康记录，按 `retention` 处理。详见[内联数据](#内联-health_context)。 |
| `mcp_servers` | array | 自带的远程 MCP 工具 `{name, url, access_token?}`。详见[自带工具](/zh/api-reference/mcp-servers)。 |
| `include_tool_steps` | bool | 返回详细的 `tool_steps`（默认 `false`）。当为 `false` 时，`tool_steps[].result` 会被截断为 2000 字符。 |

<Note>
**采样参数。** 为兼容 OpenAI，请求会接受 `max_tokens`、`n`、`temperature`、`top_p`、`stop` 和 `seed`，但智能体自行掌控其生成过程，因此这些参数实际上**不强制生效**：`max_tokens` 不会限制输出长度，`temperature` / `top_p` / `stop` / `seed` 会被忽略，且 `n` 取非 `1` 的值也**不会**报错 —— 响应仍只返回单个 choice。
</Note>

## 响应

`message.content` 携带**一段干净的最终回答**。采集阶段（思考 + 工具调用）会在单独的通道中呈现，因此回答本身永远不会被工具叙述污染。

```jsonc
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 0,
  "model": "mirobody-flash",
  "choices": [{
    "index": 0,
    "finish_reason": "stop",
    "message": {
      "role": "assistant",
      "content": "Your fasting glucose has trended down ~8% over the last 30 days ...",
      "reasoning_content": "",                                                     // deep-thinking (CoT); may be empty
      "tool_steps": [                                                              // optional, ordered
        { "id": "call_abc", "name": "search_health_indicators",
          "arguments": { "keywords": ["fasting glucose", "glucose"] },
          "result": "{...}" }
      ]
    }
  }],
  "usage": { "prompt_tokens": 0, "completion_tokens": 0, "total_tokens": 0 },
  "health_records": [ { "tool": "fetch_health_data", "data": "..." } ],
  "citations": []
}
```

| 通道 | 内容 |
| --- | --- |
| `message.content` | 仅最终回答（"答复"通道）。 |
| `message.reasoning_content` | 模型的**深度思考**（thinking / CoT）过程细节。当该层级不产出时，可能为空字符串。 |
| `message.tool_steps[]` | 工具调用轨迹 —— 每项为 `{id, name, arguments, result}`，按 `id` 合并。`arguments` 携带真实的调用入参；非流式响应上**没有 `status` 字段**。 |
| `health_records` | 回答所用到的可追溯工具输出 —— 一个 `{tool, data}` 对的数组。这里承载着当前的可解释性信息。 |
| `citations` | 预留给 `record:N` 形式的引用；**当前返回为空** —— 请依赖 `health_records`。 |
| `usage` | Token 计量。在测试集群上计数均为 `0`，且没有 `reasoning_tokens` 字段。 |

`reasoning_content` 与 `tool_steps` 都是可选的附加通道 —— 只读取 `message.content` 的客户端仍可正常工作。（非流式响应上**不存在** `message.reasoning_steps`。）

<Note>
`tool_steps[].arguments` 携带真实的工具入参（例如 `{"keywords": ["fasting glucose"]}`）。传入 `include_tool_steps: true` 可获取完整的 `result` 负载 —— 否则每个 `result` 会被截断为约 2000 字符。
</Note>

## 流式传输

当 `stream: true` 时，帧按顺序到达 —— 先是工具 / 思考增量，再是单条不中断的回答流：

```text
data: {"object":"chat.completion.chunk","choices":[{"delta":{"reasoning_content":"<fragment>"}}]}   // when the tier emits thinking
data: {"object":"chat.completion.chunk","choices":[{"delta":{"tool_steps":[{"id":"call_abc","name":"search_health_indicators","arguments":{...}}]}}]}
data: {"object":"chat.completion.chunk","choices":[{"delta":{"content":"<answer fragment>"}}]}
data: {"object":"chat.completion.chunk","choices":[{"delta":{},"finish_reason":"stop"}],"health_records":[],"citations":[]}
data: [DONE]
```

各帧都是 `chat.completion.chunk` 对象。按 `id` 合并 `tool_steps`。**最后一帧**（在 `[DONE]` 之前）携带顶层的 `health_records` / `citations`；流中**不包含** token `usage`。只读取 `delta.content` 的客户端仍可正常工作 —— 回答是一条干净的流。

## 内联 health_context

只为本轮附带数据，而不必先写入存储：

```python
resp = client.chat.completions.create(
    model="mirobody-flash",
    messages=[{"role": "user", "content": "Is this reading high?"}],
    user="alice",
    extra_body={
        "health_context": [{"indicator": "systolic_bp", "value": 148, "unit": "mmHg", "time": "2026-06-16T08:00:00Z"}],
        "retention": "none",   # used this turn, then deleted
    },
)
```

当 `retention="none"` 时，数据在请求完成时被清除 —— 不会持久化任何内容。

## 系统提示词

`system` 消息设定回答的语气、人设、格式与语言。数据采集由平台控制 —— 智能体始终读取该 Subject 的真实数据 —— 因此 `system` 提示词只塑造回答*如何*呈现，永远不会影响*是否*使用真实数据。

## 内置工具

这些工具你不必自己调用 —— 智能体会按需针对数据主体(Subject)的数据调用它们。当前托管环境的工具目录为：

- `search_health_indicators`、`fetch_health_data` —— 查找并读取数据主体(Subject)的记录
- `record_health_indicator` —— 写入一条新的结构化读数
- `list_family_members` —— 解析该数据主体(Subject)有权查询的关护圈（care-circle）成员
- `list_fhir_resources`、`write_fhir_observation` —— 读取 / 写入 FHIR 资源
- `search_medical_literature`、`get_clinical_trials`、`get_article_by_pmcid` —— 查询医学证据
- `fetch_url` —— 读取一个网页
- `ask_user` —— 暂停并提出澄清问题（在流中以中断（interrupt）的形式呈现）

具体的工具集会因集群与部署而异 —— 在你的集群上调用公开的 `GET /api/health` 即可查看其实时 `tool_names` 与 `agent_names`。你通过 [`/v1/files`](/zh/api-reference/files) 上传的文件会被智能体自动读取。要添加你自己的工具，请通过 [`mcp_servers`](/zh/api-reference/mcp-servers) 接入远程 MCP 服务器。

## 图表（vis-chart）

当回答涉及趋势 / 对比 / 分布时，可能会嵌入一个用围栏包裹的 ` ```vis-chart ` 块，内含纯数据 JSON（`{type, title, axisXTitle, axisYTitle, data}`）。**渲染是客户端的工作** —— API 只返回数据规格。请检测这个围栏块并渲染它（折线图 / 面积图 / 柱状图 / 饼图）。
