---
title: "限流"
description: "当前限流状态 + 客户端节流。"
---

**当前不强制执行每请求的硬性限流。** 生产套餐将引入按账户的配额;届时超额会返回 HTTP `429` 并附带 `Retry-After` 响应头(单位:秒),本页也会同步给出确切数字。

```text
HTTP/1.1 429 Too Many Requests
Retry-After: 47
```

<Note>
即使没有硬性限流,持续的滥用流量也可能在平台边缘被节流。现在就为客户端加上 `429` + `Retry-After` 处理——今天只是一个 `if`,以后能省一次事故。
</Note>

## 推荐节流

| 路径 | 建议 RPM | 说明 |
| --- | --- | --- |
| `POST /v1/chat/completions` | ≤ 60 | 流式调用同样计数;请等一次调用结束后再发起下一次 |
| `POST /v1/files` | ≤ 30 | 单文件上限 **100 MB**;OCR 计算开销较大 |
| `POST /v1/data` | ≤ 120 | 将记录批量合并到一次请求中(每批最多 500 条) |
| `GET` 读取(`/v1/data`、`/v1/files`、`/v1/models`) | ≤ 120 | — |

## 客户端退避

- 收到 `429` 时,**遵守 `Retry-After`**——切勿立即重试。
- 连续两次 `429` 后将退避时长翻倍,上限为 5 分钟。
- 不要并行发起 chat 调用;先让每个流式调用完成。
