## 关联如何工作

连接一个数据源分两个阶段：**配置凭据**（由运维方一次性完成）与**关联账号**（按用户、所有权门控）。`GET /vendors/{id}/data` 只返回 vendor 原生 JSON，不会存储；`POST /vendors/{id}/sync` 将支持的值映射为 FHIR Observation 并持久化。底层契约见 [Vendor 系统](/zh/concepts/providers)。

## 1 —— 配置凭据

每个 vendor 从中央配置读取其凭据，使用简洁的按 vendor 键，可写在 `config.yml` 中或用同名环境变量设置（`<ID>` 是 vendor id 的大写形式；参见 `config.example.yml`）：

```bash
<ID>_API_KEY=...        # bearer / API key（多数 REST vendor）
<ID>_CLIENT_ID=...      # OAuth client id
<ID>_CLIENT_SECRET=...  # OAuth client secret
<ID>_BASE_URL=...       # 主机覆盖；留空 => vendor 默认
```

示例：

```bash
# Fitbit（设备品牌，OAuth 2.0）
FITBIT_CLIENT_ID=your_client_id
FITBIT_CLIENT_SECRET=your_client_secret

# Terra（聚合器）
TERRA_API_KEY=your_api_key

# 通用 EHR（SMART on FHIR）—— base_url 必填（按租户的 FHIR base）
EHR_BASE_URL=https://fhir.epic.com/interconnect-fhir-oauth/api/FHIR/R4/
EHR_API_KEY=smart_on_fhir_access_token
```

<Info>
环境变量优先于 `config.yml`。Fernet 密文（`gAAAA…`）可在加载时经 `CONFIG_ENCRYPTION_KEY` 解密；密钥派生规则必须一致，请按[配置](/zh/configuration)操作，不要假设随机生成的 Fernet key 可以直接复用。旧的 `MIROBODY_VENDOR_<ID>_*` 环境变量约定已是遗留用法 —— 只有独立的 `vendor` 调试 CLI 仍在读取。
</Info>

在 vendor 的开发者门户注册你的应用以获取 client id / secret（或 API key）。通用 `ehr` 客户端需要租户的 FHIR **服务 base URL**，可从公开的 Service Base URL 目录（ONC Lantern；Oracle Health / Cerner）发现。

## 2 —— 关联账号

关联受**所有权门控**，并限定在当前 JWT 用户范围内。仓库提供两种连接路径；具体支持程度因 vendor 而异。

### 浏览器 OAuth

对于已经实现 `authorize_url` 与 token exchange 的 vendor：

1. 携带用户 bearer JWT 调用 `GET /vendors/{id}/authorize`。
2. 把浏览器重定向到返回的 `authorize_url`。
3. Vendor 重定向到公开的 `GET /vendors/callback?code=&state=`。服务端消费一次性 state、交换 code；在已配置 vendor/file 加密 key 时加密存储 token，然后把关联标记为 verified 并跳回应用。

```bash
curl "http://localhost:8080/vendors/fitbit/authorize" \
  -H "Authorization: Bearer $JWT"
```

请把 `VENDOR_REDIRECT_URI` 设置为服务端 callback URL。若某个 vendor 的 authorize 或 exchange 是显式 stub，接口会返回错误；展示连接按钮前先核对 [Provider 状态](/zh/providers/overview)。

<Warning>
使用每用户 OAuth 前，必须设置 `VENDOR_TOKEN_ENCRYPTION_KEY` 或 `FILE_ENCRYPTION_KEY`。两者都没有时，bearer token 不会以明文持久化，后续拉取会回退到共享配置凭据。见[配置 → Vendor OAuth token](/zh/configuration#vendor-oauth-token)。
</Warning>

### 手动关联或配置凭据

对于非浏览器集成，先用固定请求体创建 pending link，再通过该 vendor 支持的证明路径验证：

```bash
curl -X POST "http://localhost:8080/vendors/fitbit/bind" \
  -H "Authorization: Bearer $JWT" \
  -H "Content-Type: application/json" \
  -d '{"external_user_id":"FITBIT_USER_ID"}'

curl -X POST "http://localhost:8080/vendors/fitbit/bind/verify" \
  -H "Authorization: Bearer $JWT" \
  -H "Content-Type: application/json" \
  -d '{}'
```

`bind/verify` 可接收 `{code, redirect_uri}` 做 OAuth code exchange；空对象会走配置凭据验证。它不是通用绕过方式：不支持的 vendor 操作会明确失败。`UNIQUE(vendor_id, external_user_id)` 约束防止两个本地用户认领同一外部账号。

## 3 —— 拉取或同步

验证后，可按一个已声明的数据域拉取 vendor 原生 JSON：

```bash
curl "http://localhost:8080/vendors/fitbit/data?domain=heart_rate&start=2026-06-01&end=2026-06-30" \
  -H "Authorization: Bearer $JWT"
```

需要持久化支持的值时，使用相同查询参数调用 `POST /vendors/{id}/sync`。省略 `domain` 会尝试该 vendor 声明的全部域，并返回 `{posted, failed}`；尚无映射的 vendor payload 合法地可能写入零条 Observation。

合法 domain 为 `activity`、`sleep`、`heart_rate`、`glucose`、`nutrition`、`body_metrics`、`labs`、`clinical`。Vendor 会拒绝它不支持的域。

## 连接一个 EHR（SMART on FHIR）

EHR 是**网页**能够自行连接的唯一临床数据源（Apple / Samsung 仅设备本地可用）。Mirobody 充当该 EHR 的 OAuth **客户端**。浏览器驱动的流程位于 `/health/ehr/*` 下：

<Steps>
  <Step title="查找租户">
    在 Service Base URL 目录中搜索该医院 / 诊所：

    ```bash
    curl "http://localhost:8080/health/ehr/providers?q=Epic" -H "Authorization: Bearer $JWT"
    ```
  </Step>
  <Step title="Authorize">
    `POST /health/ehr/authorize {fhir_base_url}` 运行 SMART 发现（`/.well-known/smart-configuration`）+ PKCE，缓存 verifier/state，并返回供浏览器跳转的 `authorize_url`。
  </Step>
  <Step title="Callback">
    EHR 重定向回 `GET /health/ehr/callback?code=&state=`；该 code 会被换成一个 access token，按用户缓存。
  </Step>
  <Step title="Sync">
    `POST /health/ehr/sync` 使用缓存的 token，经 `ehr` 客户端获取 `Observation` 并逐条经 FHIR 存储持久化。
  </Step>
</Steps>

用 `SMART_FHIR_*` 系列键配置 SMART 客户端（client id / redirect uri / scope；参见 `config.example.yml`）。在 Web 客户端中，这是 **Settings → Connect EHR**。

## 设备本地存储（此处无需关联）

Apple Health、Samsung Health、Google Health Connect 以及小米在服务端**没有服务端 API，也没有关联流程**。原生宿主 App 在设备本地读取它们，并把 FHIR `Observation` POST 到 `/fhir`。参见[数据流](/zh/concepts/data-flow)。

## 自带 provider

如果你需要的数据源尚未发布，在 `src/health/vendor/` 下添加一个 `vendor::Vendor` 并注册它 —— 参见 [Provider 集成](/zh/development/provider-integration)。由于 vendor 是编译进程序的，你需要重新构建内核，而不是在运行时投放一个文件。
