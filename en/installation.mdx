title: "Installation"

Mirobody is a single **C++11** core built with **CMake** (plus vcpkg on Windows). The same source produces several artifacts: the standalone `mirobody` binary, a shared library (`libmirobody`), a Python wheel, an Android `.so`, and an iOS static library / xcframework. This page covers building each one.

<Info>
Mirobody is a native binary you build from source: run `./build.sh` (or `build.cmd` on Windows), or use Docker.
</Info>

## Standalone binary (desktop / server)

<Tabs>
  <Tab title="Linux / WSL / macOS">
    ### 1. Clone

    ```bash
    git clone https://github.com/thetahealth/mirobody.git
    cd mirobody
    ```

    ### 2. Install the native dependencies

    vcpkg is Windows-only here ‚Äî Linux and macOS use the system package manager, and CMake's `find_package` picks the libraries up directly. These are the **core** dependencies (always needed). The desktop default backend is PostgreSQL, so install `libpq` too.

    <Tabs>
      <Tab title="Debian / Ubuntu / WSL">
        ```bash
        sudo apt install build-essential cmake ninja-build pkg-config \
                         libwebsockets-dev libcurl4-openssl-dev libssl-dev \
                         rapidjson-dev libyaml-cpp-dev libhiredis-dev \
                         libjpeg-dev libpng-dev libtiff-dev libwebp-dev

        # Database client (PostgreSQL is the desktop default)
        sudo apt install libpq-dev
        ```
      </Tab>

      <Tab title="Fedora / RHEL">
        ```bash
        sudo dnf install gcc-c++ cmake ninja-build pkgconf-pkg-config \
                         libwebsockets-devel libcurl-devel openssl-devel \
                         rapidjson-devel yaml-cpp-devel hiredis-devel \
                         libjpeg-turbo-devel libpng-devel libtiff-devel libwebp-devel

        # Database client (PostgreSQL is the desktop default)
        sudo dnf install libpq-devel
        ```
      </Tab>

      <Tab title="macOS (Homebrew)">
        ```bash
        brew install cmake ninja pkg-config libwebsockets curl openssl@3 \
                     rapidjson yaml-cpp hiredis \
                     jpeg-turbo libpng libtiff webp

        # Database client (PostgreSQL is the desktop default)
        brew install libpq
        ```
      </Tab>
    </Tabs>

    <Tip>
    Install just the **one** database client matching the backend you plan to build against ‚Äî SQLite, MySQL/MariaDB, and DuckDB clients are only needed if you select those backends. See [the backend list](#database-backends).
    </Tip>

    ### 3. Build

    ```bash
    ./build.sh              # host arch + POSTGRESQL  -> build/mirobody
    ```

    The backend is a command-line token (no env var), passed in any order with `clean`:

    ```bash
    ./build.sh sqlite       # SQLITE                 -> build_sqlite/
    ./build.sh legacy       # POSTGRESQL_LEGACY      -> build_legacy/
    ./build.sh legacy clean # reconfigure from scratch
    ./build.sh -h           # full token list
    ```

    Tokens: `pg` / `postgresql`, `legacy` / `pg_legacy`, `mysql`, `sqlite`, `duckdb`, `ck` / `clickhouse` (omit for the `POSTGRESQL` default). Each backend builds into its own `build[_<backend>]` directory, so several can coexist.

    Or invoke CMake directly:

    ```bash
    cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release -DMIROBODY_DATABASE_BACKEND=POSTGRESQL
    cmake --build build
    ```
  </Tab>

  <Tab title="Windows (MSVC + vcpkg)">
    ### 1. Install Visual Studio

    Install **Visual Studio 2017 or newer** (Community is fine). In the installer, under *Workloads* tick **Desktop development with C++**, and under *Individual components* tick **C++ CMake tools for Windows** ‚Äî that component ships `ninja.exe` and a bundled vcpkg, which is exactly what `build.cmd` looks for. No separate Ninja install is needed.

    ### 2. Clone

    ```cmd
    git clone https://github.com/thetahealth/mirobody.git
    cd mirobody
    ```

    ### 3. Build

    vcpkg pulls the native deps from [`vcpkg.json`](https://github.com/thetahealth/mirobody/blob/main/vcpkg.json) automatically ‚Äî no manual installs.

    ```cmd
    build.cmd              :: host arch + POSTGRESQL   -> build\mirobody.exe
    build.cmd sqlite       :: SQLITE                   -> build_sqlite\
    build.cmd legacy clean :: reconfigure that dir
    build.cmd -h           :: full token list
    ```

    <Info>
    The `VS_DIR`, `VCPKG_ROOT`, and `NINJA` environment variables are all optional ‚Äî they only override the defaults. `vcvarsall.bat` points at the VS-bundled vcpkg by default. See the repo README for the full Windows walkthrough.
    </Info>
  </Tab>
</Tabs>

### Run

```bash
./mirobody                  # reads ./config.yml by default
./mirobody --config /path/to/config.yml
./mirobody --help
```

Copy `config.example.yml` to `config.yml` first and set an LLM key ‚Äî see [Quick Start](/en/quickstart) and [Configuration](/en/configuration). The server serves HTTP + WebSocket (and the web client from `res/htdoc`) on `HTTP_HOST:HTTP_PORT`, `0.0.0.0:8080` by default.

## Docker

The repository ships a multi-stage `Dockerfile`: stage 1 compiles the `mirobody` executable from the Ubuntu system package set; stage 2 is a slim runtime image with only the shared libraries the binary links.

```bash
git clone https://github.com/thetahealth/mirobody.git
cd mirobody

# Build (PostgreSQL backend by default)
docker build -t mirobody:latest .

# Or a different SQL backend
docker build --build-arg MIROBODY_DATABASE_BACKEND=POSTGRESQL_LEGACY -t mirobody:legacy .

# Run ‚Äî the image sets HTTP_PORT=80; mount your own config.yml
docker run -p 80:80 -v $PWD/config.yml:/app/config.yml mirobody:latest
```

The image bakes `config.example.yml` as the lowest-precedence defaults layer; real settings come from the mounted `config.yml` or environment variables. See [Docker Deployment](/en/deployment/docker) for details.

## Other build outputs

The same source tree builds several embeddable artifacts beyond the standalone binary.

<AccordionGroup>
  <Accordion title="Shared library (libmirobody) ‚Äî Java / Go / C# / Node / Rust" icon="share-nodes">
    Build the C-ABI shared library (`mirobody.dll` / `libmirobody.so` / `.dylib`) that any language with a C FFI can load:

    ```bash
    ./build-shared.sh          # -> build-shared/libmirobody.*  (build-shared.cmd on Windows)
    ```

    The C API is declared in `src/mirobody.h` (`mirobody_start` / `mirobody_stop` / `mirobody_chat` / ‚Ä¶). Runnable bindings for Go, C#, Node, Rust, and Java live under `bindings/`.
  </Accordion>

  <Accordion title="Python wheel ‚Äî built from source (not PyPI)" icon="python">
    `mirobody._mirobody` is a CPython extension (a pybind11 module wrapping `mirobody::Server`) packaged into a `.whl` by scikit-build-core ‚Äî you **build it from source**:

    ```bash
    ./build-python.sh          # Linux / macOS  (build-python.cmd on Windows)
    ```

    Then drive the embedded server in-process:

    ```python
    import mirobody
    with mirobody.Server(listen_port=8080, openai_api_key="sk-...") as srv:
        print("listening on", srv.listen_port())
    ```

    See `python/README.md` in the repo for details.
  </Accordion>

  <Accordion title="Android ‚Äî libmirobody.so via JNI" icon="android">
    The Android project under `android/` pulls the same `CMakeLists.txt` via `externalNativeBuild` and builds the `mirobody` target as a shared library (`libmirobody.so`) loaded by the host app over JNI. Native deps come from prebuilt sysroots under `android/prebuilt/<ABI>/` (`arm64-v8a` today).

    ```bash
    cd android
    ./gradlew assembleRelease
    ```
  </Accordion>

  <Accordion title="iOS ‚Äî libmirobody.a / mirobody.xcframework" icon="apple">
    The iOS target builds `libmirobody.a` exposing the C API in `src/mirobody.h`; a Swift/Objective-C host links it and calls `mirobody_start` / `mirobody_stop`. Native deps come from prebuilt sysroots under `ios/prebuilt/<sdk>-<arch>/`.

    ```bash
    # device slice
    cmake -S . -B build-ios-arm64 -G Xcode \
        -DCMAKE_TOOLCHAIN_FILE=path/to/ios.toolchain.cmake \
        -DPLATFORM=OS64 -DDEPLOYMENT_TARGET=15.0 \
        -DCMAKE_PREFIX_PATH="$(pwd)/ios/prebuilt/iphoneos-arm64"
    cmake --build build-ios-arm64 --config Release
    ```

    Bundle device + simulator slices into an xcframework with `xcodebuild -create-xcframework`. Only `arm64` slices are built.
  </Accordion>

  <Accordion title="Desktop (Electron) and Qt clients" icon="desktop">
    `electron/` embeds `libmirobody` in-process via koffi FFI and uses the self-contained **SQLite** backend (no external database) ‚Äî build the shared library and web client, then `cd electron && npm install && npm start`. `qt/` is a native Qt Quick (QML) client that is a pure HTTP/SSE API client (it does **not** embed the core) and talks to any running `mirobody` backend. Both are covered in the repo READMEs.
  </Accordion>
</AccordionGroup>

## Optional document-format libraries

The core builds fine without these; the matching format just reports "not built" at runtime. Enable them with CMake flags:

| Feature | CMake flag | Notes |
| ------- | ---------- | ----- |
| `.xlsx` reader | *(auto-detected)* | Via **xlnt**. Not packaged on Linux/macOS ‚Äî build from source; required on Windows via vcpkg. |
| PDF text + rasterization | `-DMIROBODY_ENABLE_PDF=ON` | Via **PDFium** ‚Äî download a prebuilt no-V8 release and point `-DCMAKE_PREFIX_PATH` at it. |
| OCR for scanned PDFs | `-DMIROBODY_ENABLE_OCR=ON` | Via **Tesseract** (implies PDF); `apt install tesseract-ocr libtesseract-dev libleptonica-dev tesseract-ocr-eng`. |
| Legacy `.xls` | `-DMIROBODY_ENABLE_XLS=ON` | Via **libxls** ‚Äî no package, vendor it. |

CSV is built-in (no library). See [File Processing](/en/concepts/file-processing).

## Database backends

Exactly **one** SQL backend is linked into the binary, chosen at build time via `-DMIROBODY_DATABASE_BACKEND=` (or the `build.sh` token). SQL schemas live under `res/sql/`.

| Backend value | Default for | `build.sh` token |
| ------------- | ----------- | ---------------- |
| `POSTGRESQL` | Desktop / server | `pg` / `postgresql` *(default)* |
| `SQLITE` | Mobile / self-contained | `sqlite` |
| `POSTGRESQL_LEGACY` | ‚Äî | `legacy` / `pg_legacy` |
| `MYSQL` | ‚Äî | `mysql` |
| `DUCKDB` | ‚Äî | `duckdb` |
| `CLICKHOUSE` | ‚Äî *(stub today)* | `ck` / `clickhouse` |

## Verify the installation

<Steps>
  <Step title="Health check">
    ```bash
    curl http://localhost:8080/api/health
    ```
    Returns `ok`.
  </Step>

  <Step title="MCP discovery">
    ```bash
    curl -X POST http://localhost:8080/mcp \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
    ```
    Lists the built-in tools.
  </Step>

  <Step title="Logs">
    The binary logs to stdout by default; set `LOG_DIR` in `config.yml` to write files. Under Docker, `docker logs <container>`.
  </Step>
</Steps>

## Troubleshooting

<AccordionGroup>
  <Accordion title="libwebsockets too old / websocket_client.cpp fails to compile" icon="circle-exclamation">
    The code needs libwebsockets ‚â• 4.1. Ubuntu 24.04 ships 4.3; older distros (and Debian bookworm) may be too old ‚Äî install a newer build or use the Docker image, which is based on Ubuntu 24.04.
  </Accordion>

  <Accordion title="find_package can't locate a dependency" icon="magnifying-glass">
    Re-check the platform dependency list above. If a library is installed in a non-standard prefix, point CMake at it with `-DCMAKE_PREFIX_PATH=<install-root>`.
  </Accordion>

  <Accordion title="Port conflict" icon="network-wired">
    Change `HTTP_PORT` in `config.yml` (default `8080`), then restart.
  </Accordion>

  <Accordion title="Database connection fails" icon="database">
    The desktop build links PostgreSQL ‚Äî set the `PG_*` keys in `config.yml`, or rebuild with `./build.sh sqlite` for a self-contained SQLite backend that needs no external database.
  </Accordion>
</AccordionGroup>
