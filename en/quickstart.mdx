## Installation Steps

<Steps>
  <Step title="Clone the repository">
    Clone the Mirobody Health repository to your local machine:

    ```bash
    git clone https://github.com/thetahealth/mirobody-health.git
    cd mirobody-health
    ```
  </Step>

  <Step title="Set up configuration files">
    Create your `.env` file and the config file:

    ```bash
    cp .env.example .env
    cp config.yaml config.localdb.yaml
    ```
    
    <Tip>
    The `ENV` variable tells the system which config file to use. `ENV=localdb` loads `config.localdb.yaml`.
    </Tip>
  </Step>

  <Step title="Set environment variable">
    Edit `.env` to set your configuration file:

    ```bash .env
    ENV=localdb
    ```

    This tells the system to use `config.localdb.yaml` for configuration.
  </Step>

  <Step title="Add API keys and configure agents">
    Create `config.localdb.yaml` and add at least one LLM API key:

    ```yaml config.localdb.yaml
    # ============ LLM API Keys (At least one required) ============
    GOOGLE_API_KEY: ''      # For Gemini models
    OPENAI_API_KEY: ''      # For GPT models  
    OPENROUTER_API_KEY: ''  # For Claude, DeepSeek, and other models via OpenRouter
    
    # ============ Agent Configuration ============
    # Tool directories for MCP tools
    MCP_TOOL_DIRS:
      - mirobody/pub/tools
      - mirobody/pub/tools_health
    
    # Resource directories for MCP resources  
    MCP_RESOURCE_DIRS:
      - mirobody/pub/resources
    
    # Agent directories
    AGENT_DIRS:
      - mirobody/pub/agents
    ```

    <Warning>
    **At least one API key is required** for the system to work:
    - **GOOGLE_API_KEY**: Get from [Google AI Studio](https://aistudio.google.com/app/apikey)
    - **OPENAI_API_KEY**: Get from [OpenAI Platform](https://platform.openai.com/api-keys)  
    - **OPENROUTER_API_KEY**: Get from [OpenRouter](https://openrouter.ai/keys) - provides access to Claude, DeepSeek, and other models
    </Warning>
    
    <Info>
    Mirobody Health includes two agent types:
    - **DeepAgent** (LangChain-based): Best for complex tasks, file operations, multi-step planning
    - **BaselineAgent** (Native Gemini): Best for simple conversations, lightweight MCP integration
    
    See [Tools Overview](/tools/overview) for details on agent capabilities and configuration.
    </Info>
  </Step>

  <Step title="Configure providers (Optional)">
    If you want to connect wearable devices like Garmin or Whoop, add their OAuth credentials:

    ```yaml config.localdb.yaml
    # Provider OAuth Credentials (Optional)
    GARMIN_CLIENT_ID: ''
    GARMIN_CLIENT_SECRET: ''
    GARMIN_REDIRECT_URI: ''
    
    WHOOP_CLIENT_ID: ''
    WHOOP_CLIENT_SECRET: ''
    WHOOP_REDIRECT_URI: ''
    ```

    See the [Configuration guide](/configuration) for complete provider setup.
  </Step>

  <Step title="Deploy with Docker">
    Run the deployment script to start all services:

    ```bash
    ./run.sh
    ```

    This will start three services:
    - **Backend API** on port 18080
    - **PostgreSQL** database on port 5432
    - **Redis** cache on port 6379

    <Check>
    Wait for all services to start. You should see logs indicating successful startup.
    </Check>
  </Step>

  <Step title="Verify the installation">
    Open your browser and navigate to:

    ```
    http://localhost:18080
    ```

    You should see the Mirobody Health welcome page.
  </Step>

  <Step title="Log in">
    Log in with the default demo account or the accounts set up in `config.yaml`.

    ``` 
    demo1@mirobody.ai
    777777
    ```

    Once logged in, you can:
    - **Upload files**: Lab reports, medical records, health documents (see [File Processing](/en/concepts/file-processing))
    - **Chat with AI**: Ask questions about your health data
    - **Connect providers**: Link Garmin, Whoop, and other devices (see [Using Providers](/en/providers/using-providers))
  </Step>
</Steps>

## Test the API

Once the services are running, you can test the API endpoints:

<Tabs>
  <Tab title="Health Check">
    Verify the server is running:

    ```bash
    curl http://localhost:18080/health
    ```
  </Tab>

  <Tab title="List Providers">
    Get available health providers:

    ```bash
    curl http://localhost:18080/api/v1/pulse/providers
    ```
  </Tab>

  <Tab title="MCP Interface">
    Test the MCP protocol endpoint:

    ```bash
    curl -X POST http://localhost:18080/mcp \
      -H "Content-Type: application/json" \
      -d '{
        "jsonrpc": "2.0",
        "id": 1,
        "method": "tools/list"
      }'
    ```
  </Tab>
</Tabs>

## Docker Services

The deployment includes three containerized services:

| Service | Port | Description |
|---------|------|-------------|
| backend | 18080 | FastAPI application server |
| postgres | 5432 | PostgreSQL database |
| redis | 6379 | Redis cache and session store |

<Tip>
You can view logs for any service using:

```bash
docker-compose logs -f backend
docker-compose logs -f postgres
docker-compose logs -f redis
```
</Tip>

    docker-compose exec redis redis-cli

  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Port already in use" icon="circle-exclamation">
    If port 18080 is already in use, edit `config.yaml`:

    ```yaml
    HTTP_PORT: 18081  # Change to an available port
    ```

    Then restart the services.
  </Accordion>

  <Accordion title="Database connection failed" icon="database">
    Ensure PostgreSQL container is running:

    ```bash
    docker-compose ps postgres
    ```

    Check database logs:

    ```bash
    docker-compose logs postgres
    ```
  </Accordion>

  <Accordion title="Redis connection failed" icon="server">
    Verify Redis is running and accessible:

    ```bash
    docker-compose exec redis redis-cli ping
    ```

    Should return `PONG` if working correctly.
  </Accordion>

  <Accordion title="Permission denied on deploy.sh" icon="lock">
    Make the script executable:

    ```bash
    chmod +x deploy.sh
    ```
  </Accordion>
</AccordionGroup>
