<Warning>
**Whoop is not a shipped vendor client.** The confirmed device-brand clients in `src/health/vendor/device/` are **Fitbit, Garmin, and Withings** only. Whoop has no first-party `vendor::Vendor` — it would be reached through an **aggregator platform** (e.g. Terra, Rook, Spike), or you'd add a client yourself. *(verify — Whoop may only be reachable via an aggregator.)* This page therefore uses **Fitbit** as the OAuth 2.0 example, since it's a real, fully-implemented device client.
</Warning>

## Overview

**Fitbit** ([`src/health/vendor/device/fitbit.cpp`](https://github.com/thetahealth/mirobody/blob/main/src/health/vendor/device/fitbit.cpp), id `fitbit`) is the best example of a device brand with a **complete** OAuth 2.0 implementation against the vendor contract: it implements `fetch`, `authorize_url`, `revoke`, and a signed `handle_webhook`. Contrast it with the [Garmin example](/en/examples/garmin-provider), which is mostly stubs because Garmin's data API is partner-gated.

<Info>
This is a `vendor::Vendor`. See [Vendor System](/en/concepts/providers) for the interface it implements.
</Info>

## `authorize_url` — build the consent URL

A confidential server client authenticates with its secret at the token step, so **PKCE is not used**; `authorize_url` is pure string construction (no network call), requiring the OAuth `client_id`:

```cpp
std::string authorize_url(const std::string& redirect_uri, const std::string& state,
                          const std::string& /*user_id*/, const std::string& /*provider*/) override {
    if (config().client_id.empty()) {
        throw VendorError(info_.id + ": authorize_url requires an OAuth client_id — set "
                          "MIROBODY_VENDOR_FITBIT_CLIENT_ID");
    }
    std::string url = std::string(kAuthorizeUrl) +   // https://www.fitbit.com/oauth2/authorize
                      "?response_type=code&client_id=" + url_encode(config().client_id) +
                      "&scope=" + url_encode(kDefaultScope);  // "activity heartrate sleep weight profile"
    if (!redirect_uri.empty()) url += "&redirect_uri=" + url_encode(redirect_uri);
    if (!state.empty())        url += "&state=" + url_encode(state);
    return url;
}
```

The token exchange itself happens out of band at `/oauth2/token`; the resulting access token is stored as `config.api_key`.

## `fetch` — per-domain time-series GETs

Fitbit's Web API serves whole-day ranges (`yyyy-MM-dd`). `fetch` maps a `DataDomain` onto the documented endpoint and returns Fitbit's JSON:

```cpp
static std::string endpoint(DataDomain d, const std::string& u,
                            const std::string& s, const std::string& e) {
    switch (d) {
        case DataDomain::Activity:    return "/1/user/" + u + "/activities/steps/date/" + s + "/" + e + ".json";
        case DataDomain::HeartRate:   return "/1/user/" + u + "/activities/heart/date/" + s + "/" + e + ".json";
        case DataDomain::Sleep:       return "/1.2/user/" + u + "/sleep/date/" + s + "/" + e + ".json";
        case DataDomain::BodyMetrics: return "/1/user/" + u + "/body/log/weight/date/" + s + "/" + e + ".json";
        default:
            throw VendorError("fitbit: unsupported domain (brokers: activity, heart_rate, sleep, body_metrics)");
    }
}
```

The user path segment is the Fitbit user id, or `-` (the API's alias for the token owner). The access token (`config.api_key`) rides as a bearer header.

## `handle_webhook` — verify the signature

Fitbit signs subscription notifications with `X-Fitbit-Signature` = `base64(HMAC-SHA1(raw_body))` keyed by the client secret + `"&"`. The client verifies it with a constant-time compare before parsing:

```cpp
const std::string key = config().client_secret + "&";
std::array<unsigned char, 20> mac = storage::hmac_sha1(key, body);
const std::string expected = storage::base64_encode(mac.data(), mac.size());
if (!ct_equal(expected, sig)) {
    throw VendorError(info_.id + ": webhook signature verification failed (X-Fitbit-Signature)");
}
```

Because the signing scheme is **publicly documented**, this can be implemented faithfully — the reason Fitbit's webhook is real where Garmin's is a stub.

## `revoke` — drop the grant

`revoke` POSTs the token to `/oauth2/revoke`, authenticated as the confidential client (HTTP Basic `client_id:client_secret`).

## Configuration

```bash
MIROBODY_VENDOR_FITBIT_CLIENT_ID=your_client_id
MIROBODY_VENDOR_FITBIT_CLIENT_SECRET=your_client_secret
# after the OAuth token exchange, the access token is used as the API key:
MIROBODY_VENDOR_FITBIT_API_KEY=fitbit_oauth2_access_token
# optional host override (default https://api.fitbit.com)
MIROBODY_VENDOR_FITBIT_BASE_URL=https://api.fitbit.com
```

## If you really need Whoop

Two honest paths:

1. **Via an aggregator** — connect Whoop through Terra / Rook / Spike (whichever lists Whoop in its provider catalogue) and fetch through that aggregator's `vendor::Vendor`. *(verify the aggregator's current Whoop support.)*
2. **Write a first-party client** — add `src/health/vendor/device/whoop.cpp` implementing the contract against Whoop's public OAuth 2.0 API, and register it. Follow [Provider Integration](/en/development/provider-integration); Fitbit above is the closest template.
