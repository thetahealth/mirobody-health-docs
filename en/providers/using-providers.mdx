## How linking works

Connecting a source has two phases: **configure credentials** (once, by the operator) and **link an account** (per user, ownership-gated). `GET /vendors/{id}/data` fetches vendor-native JSON without storing it; `POST /vendors/{id}/sync` maps supported values to FHIR Observations and persists them. For the underlying contract see [Vendor System](/en/concepts/providers).

## 1 ŌĆö Configure credentials

Each vendor reads its credentials from the central config, using clean per-vendor keys settable in `config.yml` or as same-named environment variables (`<ID>` is the vendor id upper-cased; see `config.example.yml`):

```bash
<ID>_API_KEY=...        # bearer / API key (most REST vendors)
<ID>_CLIENT_ID=...      # OAuth client id
<ID>_CLIENT_SECRET=...  # OAuth client secret
<ID>_BASE_URL=...       # host override; empty => vendor default
```

Examples:

```bash
# Fitbit (device brand, OAuth 2.0)
FITBIT_CLIENT_ID=your_client_id
FITBIT_CLIENT_SECRET=your_client_secret

# Terra (aggregator)
TERRA_API_KEY=your_api_key

# Generic EHR (SMART on FHIR) ŌĆö base_url is REQUIRED (per-tenant FHIR base)
EHR_BASE_URL=https://fhir.epic.com/interconnect-fhir-oauth/api/FHIR/R4/
EHR_API_KEY=smart_on_fhir_access_token
```

<Info>
Environment variables take precedence over `config.yml`. Fernet ciphertext (`gAAAAŌĆ”`) can be decrypted on load via `CONFIG_ENCRYPTION_KEY`; its key-derivation rules matter, so follow [Configuration](/en/configuration) rather than assuming a generated Fernet key can be used directly. The older `MIROBODY_VENDOR_<ID>_*` env convention is legacy ŌĆö only the standalone `vendor` debug CLI still reads it.
</Info>

Register your application on the vendor's developer portal to obtain the client id / secret (or API key). The generic `ehr` client needs the tenant's FHIR **service base URL**, discoverable from the public Service Base URL directories (ONC Lantern; Oracle Health / Cerner).

## 2 ŌĆö Link an account

Linking is **ownership-gated** and scoped to the authenticated JWT user. The repository exposes two connection paths; support varies by vendor.

### Browser OAuth

For vendors with `authorize_url` and token exchange implemented:

1. Call `GET /vendors/{id}/authorize` with the user's bearer JWT.
2. Redirect the browser to the returned `authorize_url`.
3. The vendor redirects to public `GET /vendors/callback?code=&state=`. The server consumes the single-use state, exchanges the code, stores encrypted tokens when a vendor/file encryption key is configured, marks the link verified, and redirects back to the app.

```bash
curl "http://localhost:8080/vendors/fitbit/authorize" \
  -H "Authorization: Bearer $JWT"
```

Set `VENDOR_REDIRECT_URI` to the server callback URL. A vendor whose authorize or exchange operation is an explicit stub returns an error; check the [provider status](/en/providers/overview) before exposing a connect button.

<Warning>
Set `VENDOR_TOKEN_ENCRYPTION_KEY` or `FILE_ENCRYPTION_KEY` before using per-user OAuth. Without one, bearer tokens are never persisted in plaintext and later fetches fall back to shared configured credentials. See [Configuration ŌåÆ Vendor OAuth tokens](/en/configuration#vendor-oauth-tokens).
</Warning>

### Manual or configured-credential link

For non-browser integrations, create a pending link with the fixed body below, then verify it using the vendor-specific proof path:

```bash
curl -X POST "http://localhost:8080/vendors/fitbit/bind" \
  -H "Authorization: Bearer $JWT" \
  -H "Content-Type: application/json" \
  -d '{"external_user_id":"FITBIT_USER_ID"}'

curl -X POST "http://localhost:8080/vendors/fitbit/bind/verify" \
  -H "Authorization: Bearer $JWT" \
  -H "Content-Type: application/json" \
  -d '{}'
```

`bind/verify` accepts `{code, redirect_uri}` for an OAuth code exchange; an empty body asks the configured-credential verification path to prove access. It is not a universal shortcut: unsupported vendor operations fail explicitly. A `UNIQUE(vendor_id, external_user_id)` constraint prevents two local users from claiming the same vendor account.

## 3 ŌĆö Fetch or sync

After verification, fetch one advertised domain as vendor-native JSON:

```bash
curl "http://localhost:8080/vendors/fitbit/data?domain=heart_rate&start=2026-06-01&end=2026-06-30" \
  -H "Authorization: Bearer $JWT"
```

To persist supported values, call `POST /vendors/{id}/sync` with the same query parameters. Omitting `domain` attempts every domain the vendor advertises and returns `{posted, failed}`; an unmapped vendor payload may legitimately post zero Observations.

Valid domain names are `activity`, `sleep`, `heart_rate`, `glucose`, `nutrition`, `body_metrics`, `labs`, and `clinical`. A vendor rejects domains it does not broker.

## Connecting an EHR (SMART on FHIR)

An EHR is the one clinical source a **web page** can connect on its own (Apple / Samsung are on-device-only). Mirobody acts as an OAuth **client** of the EHR. The browser-driven flow lives under `/health/ehr/*`:

<Steps>
  <Step title="Find the tenant">
    Search the Service Base URL directories for the hospital / clinic:

    ```bash
    curl "http://localhost:8080/health/ehr/providers?q=Epic" -H "Authorization: Bearer $JWT"
    ```
  </Step>
  <Step title="Authorize">
    `POST /health/ehr/authorize {fhir_base_url}` runs SMART discovery (`/.well-known/smart-configuration`) + PKCE, caches the verifier/state, and returns the `authorize_url` for the browser to redirect to.
  </Step>
  <Step title="Callback">
    The EHR redirects back to `GET /health/ehr/callback?code=&state=`; the code is exchanged for an access token, cached per user.
  </Step>
  <Step title="Sync">
    `POST /health/ehr/sync` uses the cached token to fetch `Observation`s via the `ehr` client and persists each through the FHIR store.
  </Step>
</Steps>

Configure the SMART client with the `SMART_FHIR_*` keys (client id / redirect uri / scope; see `config.example.yml`). In the web client this is **Settings ŌåÆ Connect EHR**.

## On-device stores (no linking here)

Apple Health, Samsung Health, Google Health Connect, and Xiaomi have **no server API and no link flow** on the server. The native host apps read them on-device and POST FHIR `Observation`s to `/fhir`. See [Data Flow](/en/concepts/data-flow).

## Bring your own provider

If the source you need isn't shipped, add a `vendor::Vendor` in `src/health/vendor/` and register it ŌĆö see [Provider Integration](/en/development/provider-integration). Since vendors are compiled in, you rebuild the core rather than dropping a file at runtime.
