## How linking works

Connecting a source has two phases: **configure credentials** (once, by the operator) and **link an account** (per user, ownership-gated). Data then flows into the embedded FHIR store, where the agent reads it. For the underlying contract see [Vendor System](/en/concepts/providers).

## 1 — Configure credentials

Each vendor reads its credentials from environment variables, by convention (`<ID>` is the vendor id upper-cased):

```bash
MIROBODY_VENDOR_<ID>_API_KEY=...        # bearer / API key (most REST vendors)
MIROBODY_VENDOR_<ID>_CLIENT_ID=...      # OAuth client id
MIROBODY_VENDOR_<ID>_CLIENT_SECRET=...  # OAuth client secret
MIROBODY_VENDOR_<ID>_BASE_URL=...       # host override; empty => vendor default
```

Examples:

```bash
# Fitbit (device brand, OAuth 2.0)
MIROBODY_VENDOR_FITBIT_CLIENT_ID=your_client_id
MIROBODY_VENDOR_FITBIT_CLIENT_SECRET=your_client_secret

# Terra (aggregator)
MIROBODY_VENDOR_TERRA_API_KEY=your_api_key

# Generic EHR (SMART on FHIR) — base_url is REQUIRED (per-tenant FHIR base)
MIROBODY_VENDOR_EHR_BASE_URL=https://fhir.epic.com/interconnect-fhir-oauth/api/FHIR/R4/
MIROBODY_VENDOR_EHR_API_KEY=smart_on_fhir_access_token
```

<Info>
Because config values whose names contain `_KEY` / `_SECRET` / `_TOKEN` are auto-encrypted on load (Fernet, via `CONFIG_ENCRYPTION_KEY`), you can also set these in `config.yml`. Environment variables take precedence. See [Configuration](/en/configuration).
</Info>

Register your application on the vendor's developer portal to obtain the client id / secret (or API key). The generic `ehr` client needs the tenant's FHIR **service base URL**, discoverable from the public Service Base URL directories (ONC Lantern; Oracle Health / Cerner).

## 2 — Link an account (bind → verify → data)

Linking is **ownership-gated**: a user proves they own the external account before any data is read, so a self-asserted id can never reach someone else's records. The health module exposes three routes (all require a bearer JWT and are scoped to the authenticated user):

<Steps>
  <Step title="Bind">
    Record the external account id as a PENDING link.

    ```bash
    curl -X POST "http://localhost:8080/vendors/fitbit/bind" \
      -H "Authorization: Bearer $JWT" \
      -H "Content-Type: application/json" \
      -d '{"external_user_id": "FITBIT_USER_ID"}'
    ```

    <Note>The exact request body per vendor may differ; check the vendor's `.cpp` and `src/health/vendor_service.*`. *(verify)*</Note>
  </Step>

  <Step title="Verify">
    Prove ownership through the vendor's consent / claim flow. On success the link is marked VERIFIED.

    ```bash
    curl -X POST "http://localhost:8080/vendors/fitbit/bind/verify" \
      -H "Authorization: Bearer $JWT"
    ```

    A `UNIQUE(vendor_id, external_user_id)` constraint is the backstop — two users can't both claim one account.
  </Step>

  <Step title="Fetch data">
    Once verified, pull a data domain over a time range. `Vendor::fetch` runs and returns the source's JSON.

    ```bash
    curl "http://localhost:8080/vendors/fitbit/data?domain=heart_rate&start=2026-06-01&end=2026-06-30" \
      -H "Authorization: Bearer $JWT"
    ```

    Valid `domain` values: `activity`, `sleep`, `heart_rate`, `glucose`, `nutrition`, `body_metrics`, `labs`, `clinical` (a vendor rejects domains it doesn't broker).
  </Step>
</Steps>

## Connecting an EHR (SMART on FHIR)

An EHR is the one clinical source a **web page** can connect on its own (Apple / Samsung are on-device-only). Mirobody acts as an OAuth **client** of the EHR. The browser-driven flow lives under `/health/ehr/*`:

<Steps>
  <Step title="Find the tenant">
    Search the Service Base URL directories for the hospital / clinic:

    ```bash
    curl "http://localhost:8080/health/ehr/providers?q=Epic" -H "Authorization: Bearer $JWT"
    ```
  </Step>
  <Step title="Authorize">
    `POST /health/ehr/authorize {fhir_base_url}` runs SMART discovery (`/.well-known/smart-configuration`) + PKCE, caches the verifier/state, and returns the `authorize_url` for the browser to redirect to.
  </Step>
  <Step title="Callback">
    The EHR redirects back to `GET /health/ehr/callback?code=&state=`; the code is exchanged for an access token, cached per user.
  </Step>
  <Step title="Sync">
    `POST /health/ehr/sync` uses the cached token to fetch `Observation`s via the `ehr` client and persists each through the FHIR store.
  </Step>
</Steps>

Configure the SMART client with the `SMART_FHIR_*` keys (client id / redirect uri / scope; see `config.example.yml`). In the web client this is **Settings → Connect EHR**.

## On-device stores (no linking here)

Apple Health, Samsung Health, Google Health Connect, and Xiaomi have **no server API and no link flow** on the server. The native host apps read them on-device and POST FHIR `Observation`s to `/fhir`. See [Data Flow](/en/concepts/data-flow).

## Bring your own provider

If the source you need isn't shipped, add a `vendor::Vendor` in `src/health/vendor/` and register it — see [Provider Integration](/en/development/provider-integration). Since vendors are compiled in, you rebuild the core rather than dropping a file at runtime.
