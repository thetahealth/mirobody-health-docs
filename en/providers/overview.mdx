## What providers are

A **provider** (a `vendor::Vendor` in the code) is a compiled-in C++ client for one health-data source. Every provider implements the same `authorize / fetch / webhook` contract and is resolved at runtime by **id**. For the architecture — the contract, honest stubs, the registry — see [Vendor System](/en/concepts/providers).

This page lists the sources that actually ship in `src/health/vendor/`, in four buckets.

## Aggregator platforms (`platform/`)

Fifteen B2B health-data aggregators that broker wearable, lab, and/or clinical data behind one contract. Each carries real `VendorInfo` metadata (positioning, region, compliance, data domains) queryable without credentials; transport is implemented against each platform's **public** API, with undocumented operations left as explicit stubs.

| Vendor (id) | Region | Notes |
|-------------|--------|-------|
| **Terra** (`terra`) | Global | Connect widget + HMAC webhooks; broad wearable coverage |
| **Validic** (`validic`) | US | Enterprise PGHD; EHR write-back gated |
| **Human API** (`human_api`) | US | Clinical + wearable aggregation |
| **Junction** (`junction`) | US | Wearables + US lab network |
| **Metriport** (`metriport`) | US | Open-source; consolidated FHIR + devices |
| **Rook** (`rook`) | Global | Wearable API; per-source revoke |
| **Spike** (`spike`) | Global | Wearables + Nutrition-AI |
| **Thryve** (`thryve`) | EU | EU-hosted medical-grade wearable API |
| **WeFitter** (`wefitter`) | EU | Gamification; per-profile connections |
| **Vitalera** (`vitalera`) | Global | FHIR-capable; HMAC webhooks |
| **Open Wearables** (`open_wearables`) | Self-hosted | Open-source; `base_url` required |
| **Redox** (`redox`) | US | EHR interoperability; FHIR R4 |
| **Particle Health** (`particle_health`) | US | Clinical records via FHIR `$everything` |
| **HealthConnect** (`healthconnect`) | US | EHR + wearable; FHIR `Observation` |
| **LexisNexis** (`lexisnexis`) | US | Life-insurance underwriting; sales-gated |

<Info>
Implementation depth varies **per operation** — some vendors implement `fetch` + `authorize_url` + `handle_webhook` + `revoke`, others only `fetch`. The per-vendor status table lives in [`src/health/vendor/README.md`](https://github.com/thetahealth/mirobody/blob/main-v2/src/health/vendor/README.md).
</Info>

  </Card>
  <Card title="Oura" icon="ring">

  </Card>
  <Card title="Whoop" icon="watch">
    `whoop` — WHOOP Developer Platform v2, OAuth 2.0. `fetch` (`activity/sleep`, `recovery` [HR/HRV], `cycle`) and `authorize_url` implemented.

<Note>
Six pull-based brands ship first-party `fetch` implementations; supported OAuth operations vary by brand. Garmin is different: its partner API is push-based and only `revoke` is wired in this repository. Check each card and the source before designing a connection flow.
</Note>

## Clinical EHRs (`ehr/`) — SMART on FHIR

A single generic client (`ehr`) targets **SMART App Launch + FHIR R4** servers, parameterized by each tenant's FHIR base URL. Many US certified-EHR deployments expose this surface, but a working integration still depends on the tenant endpoint, app registration, scopes, and resources the organization enables.

- `base_url` is **required and per-tenant** (each hospital has its own FHIR service base URL); the client refuses to guess one.
- Auth is a SMART-on-FHIR OAuth 2.0 bearer token; `fetch` issues standard FHIR R4 `Observation` searches (`patient` + `category` + `date` range) and returns FHIR JSON.
- `EhrDirectory` can search public **Service Base URL** directories ([`ehr/directory.hpp`](https://github.com/thetahealth/mirobody/blob/main-v2/src/health/vendor/ehr/directory.hpp)): **ONC Lantern** and Oracle Health / Cerner's published bundle. Discovery does not guarantee authorization or resource availability.

The browser-driven connect flow lives at `/health/ehr/*` — see [Using Providers](/en/providers/using-providers).

## Phone health stores (`phone/`)

| Vendor (id) | Notes |
|-------------|-------|
| **Huawei** (`huawei`) | Huawei Health Kit **Cloud** REST API (Account Kit OAuth). `fetch` via `sampleSet:polymerize`; consent OAuth and subscription webhooks are stubs. |

<Note>
**Apple Health, Samsung Health, Google Health Connect, and Xiaomi / Mi Fitness are on-device-only** — they expose no server API, so they are not vendor clients. The host apps read them on-device and POST FHIR `Observation`s to the embedded FHIR endpoint. See [Data Flow](/en/concepts/data-flow).
</Note>

## Data domains

Every vendor declares which normalized `DataDomain`s it brokers, and `fetch` takes one:

`Activity` · `Sleep` · `HeartRate` · `Glucose` · `Nutrition` · `BodyMetrics` (weight, BMI, blood pressure, SpO2, …) · `Labs` · `Clinical`
