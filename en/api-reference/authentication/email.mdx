---
title: "Email Login"
description: "/email/login + /email/verify two-step flow returning a long-lived JWT access_token"
---

China cluster uses an email + 6-digit verification code flow. OAuth 2.0 is also supported — see [OAuth](/en/api-reference/authentication/oauth).

## 1. Request the code

```http
POST /email/login
Content-Type: application/json
```

```json
{ "email": "user@example.com" }
```

```json
{ "success": true, "code": 0, "msg": "ok", "data": { "email": "user@example.com" } }
```

Repeating within 60s on the same email is rejected (HTTP 429).

## 2. Verify and exchange for token

```http
POST /email/verify
Content-Type: application/json
```

```json
{ "email": "user@example.com", "code": "000000" }
```

Response (excerpt):

```json
{
  "success": true, "code": 0, "msg": "ok",
  "data": {
    "access_token":  "eyJhbGci...",
    "token_type":    "Bearer",
    "expires_in":    2592000,
    "refresh_token": "eyJhbGci...",
    "webauthn_registered": false
  }
}
```

- `access_token` is a **30-day** JWT. Refresh ahead of expiry — see [Token Refresh](/en/api-reference/authentication/refresh).
- `refresh_token` is valid for **60 days**.
- JWT `sub` claim equals the numeric `user_id`.

## 3. Carry the token

```text
Authorization: Bearer <access_token>
```

WebSocket (browsers cannot set headers on WS):

```text
?token=<access_token>
```

## 4. Account management

```http
POST /user/del              # delete the current account
POST /user/update_name      # body: { "name": "New name" }
```

## Error codes

| HTTP | code | Meaning |
| --- | --- | --- |
| 200 | -1 | Email service not configured |
| 200 | -2 | Wrong code / email validation failed |
| 200 | -3 | Internal error |
| 429 | - | Rate-limited |
