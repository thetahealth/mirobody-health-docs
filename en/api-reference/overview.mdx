Mirobody's hosted API is **OpenAI-compatible**. Point any OpenAI SDK at the base URL below, pass an `mb_live_*` key, and call the [Answers API](/en/api-reference/chat) (`/v1/chat/completions`) or the [Agent API](/en/api-reference/responses) (`/v1/responses`) ‚Äî the agent answers from each end-user's **real, standardized health data**, with a traceable tool trace. Not sure which surface? See [Choose your API](/en/api-reference/choose-your-api). The same engine also runs **self-hosted** ‚Äî build the [open-source C++ engine](/en/index) with `./build.sh` and run it yourself; the hosted API adds managed storage, keys, and billing on top.

You create keys, watch usage, and try an interactive Playground in the [developer console](https://test-platform.mirobody.ai/). These docs are the API reference.

## Authentication

<AuthKey />

| Surface | Scheme | How to get it |
| --- | --- | --- |
| `/v1/*` (the API) | `Authorization: Bearer mb_live_*` | Create a key in the [console](https://test-platform.mirobody.ai/developer/keys) ‚Üí **API Keys**. The secret is shown **once**. |
| `/api/console/*` (console) | Session **JWT** (login) | Email-code or WeChat login in the console. |

`/v1` keys are **not** JWTs ‚Äî they're long-lived API keys scoped to your account. Never ship them in client-side code.

## OpenAI compatibility

Standard OpenAI fields work as-is; Mirobody extensions ride along in `extra_body` (Python SDK) or as plain top-level JSON (curl/fetch).

| | Fields |
| --- | --- |
| **Standard** | `model`, `messages` / `input`, `stream`, `user` ‚Äî plus, on the [Agent API](/en/api-reference/responses): `instructions`, `store`, `previous_response_id`, `tools`, `tool_choice` |
| **Mirobody extensions** | `retention` (data lifetime ‚Äî see below), `session_id` (session scoping; on [`/v1/responses`](/en/api-reference/responses) it binds a durable conversation) |
| **Response extensions** | `reasoning_content` / `reasoning` items, `tool_steps[]` (server tool trace), top-level `health_records[]` / `citations[]` |

<Note>
**Sampling parameters are accepted for compatibility, but the agent runs its own generation.** `max_tokens` is accepted but **not enforced**; `temperature`, `top_p`, `stop`, and `seed` are accepted but **ignored**. On the Answers API, `tools` / `tool_choice` / `response_format` / `n>1` are **explicitly rejected with `400`** ‚Äî see [Unsupported parameters](/en/api-reference/chat#unsupported-parameters).
</Note>

## Multi-tenancy: the `user` field

Every request carries a `user` string ‚Äî the **tenant-isolation key**. The backend maps `(your account, user)` to an internal **Subject**; each `user` you pass is fully isolated from the others. Pass each end-user's stable id as `user` and their data never crosses over. Omit it and the call falls back to your account's default Subject. Subjects are not consumer accounts ‚Äî they're invisible to the Mirobody app and to other developers.

## Data retention

Anything you write to the data plane (structured records, uploaded files, stored extractions) carries a `retention` that decides how long it's kept:

| `retention` | Meaning | Lifetime |
| --- | --- | --- |
| `permanent` (alias `persistent`) | Kept until explicitly deleted | Until `DELETE /v1/data` / `DELETE /v1/files/{key}` / `DELETE /v1/subjects/{user}` |
| `session` | Bound to a `session_id` | Until `DELETE /v1/sessions/{id}`, capped at 24h |
| `1d` | Auto-deleted within a day | ‚â§ 24h |
| `6h` / `2h` / `1h` | Auto-deleted after the grain | ‚â§ 6h / 2h / 1h |

On **[`POST /v1/data`](/en/api-reference/data) `retention` is required** ‚Äî no default; omitting it (or any value outside the enum) returns `400` (`code: invalid_retention`). `retention=session` additionally requires a `session_id`. On [`POST /v1/extract`](/en/api-reference/extract) it's required when `store=true`; on [`POST /v1/files`](/en/api-reference/files) it's optional.

There is **no `retention: "none"`**. For use-and-forget analysis, run [`POST /v1/extract`](/en/api-reference/extract) with `store=false` (dry-run ‚Äî nothing persisted), or write with `retention: "1h"`. The agent **only ever reads the Subject's currently-unexpired data**.

Conversation persistence on the Agent API is a separate knob (`store`) ‚Äî see [State & memory](/en/api-reference/state-and-memory).

## Evidence: health_records & citations

Answers are **explainable and checkable** ‚Äî not "sounds plausible," but "this conclusion came from that record of yours." Both API surfaces return two top-level evidence arrays of `{tool, data}` pairs:

- **`health_records`** ‚Äî outputs of the health-data tools the answer relied on (the Subject's actual records).
- **`citations`** ‚Äî outputs of the literature tools (`search_medical_literature`, `get_clinical_trials`, `get_article_by_pmcid`); empty when no external evidence was consulted.

The full server tool trace (every call with arguments and results) is in the `tool_steps` extension.

## Error format

Errors use the OpenAI-style envelope:

```json
{
  "error": {
    "message": "`retention` is required.",
    "type": "invalid_request_error",
    "code": "invalid_retention",
    "param": "retention"
  }
}
```

`type` is consistently `invalid_request_error` for caller mistakes; `code` and `param` identify the specific problem. Observed (status, code) pairs:

| HTTP | `code` | When |
| --- | --- | --- |
| `400` | `invalid_retention`, `invalid_session`, ‚Ä¶ | Malformed request / missing required field ‚Äî `param` names the field |
| `400` | `unsupported_parameter` | A parameter this surface rejects (e.g. `tools` on the Answers API) ‚Äî `param` names it |
| `401` | `null` | Missing or malformed `Authorization` header |
| `401` | `invalid_api_key` | Bad or revoked `mb_live_*` key |
| `404` | ‚Äî | Unknown resource (`file_key` / `response_id` / `previous_response_id` / subject) |
| `413` | ‚Äî | Upload exceeds the size limit |
| `422` | ‚Äî | File text extraction failed (`/v1/extract`) |
| `429` | ‚Äî | Rate limit ‚Äî see [Rate Limits](/en/api-reference/rate-limits) |
| `500` | `internal_error` | Unexpected server-side failure |
| `502` | ‚Äî | Upstream agent error ‚Äî transient; retry with backoff |

In **streaming**, an upstream failure arrives as an SSE error frame (`{"error": ...}` on the Answers API, `response.failed` on the Agent API) before the stream ends.
