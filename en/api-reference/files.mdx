---
title: "Files"
description: "POST /v1/files, GET /v1/files — upload & parse reports, list, fetch parsed text."
---

Upload a health document and Mirobody parses it into the Subject's store. PDFs and images go through **OCR** (original text kept for `read_file`); Excel / CSV are parsed structurally (long/wide tables auto-detected) into health records. The chat agent then reads files with its `read_file` / `ls` tools.

## Upload & parse

```http
POST /v1/files
Authorization: Bearer mb_live_*
Content-Type:  multipart/form-data
```

| Field | Description |
| --- | --- |
| `file` | The document (PDF / image / Excel / CSV). |
| `user` | Subject the file belongs to. |
| `retention` | `permanent` (default) / `1h` / `2h` / `6h` / `1d` / `session`. The hour grains auto-delete that many hours after write (still hard-capped at ≤ 24h). |

```bash
curl https://mcp.thetahealth.cn/v1/files \
  -H "Authorization: Bearer $MIROBODY_API_KEY" \
  -F "user=alice" \
  -F "file=@checkup_2026.pdf"
```

Response:

```json
{
  "file_key": "uploads/20260616_xxx.pdf",
  "filename": "checkup_2026.pdf",
  "indicators": 3,
  "abstract": "Annual checkup — glucose slightly elevated ...",
  "subject": "alice"
}
```

`indicators` is an **integer count** of structured indicators extracted, not a list of names.

<Note>
Images and PDFs are OCR'd to text that the agent can `read_file`; structured-indicator extraction is asynchronous / best-effort, so the upload response's `indicators` is often `0` for images. For guaranteed structured data, use [`POST /v1/data`](/en/api-reference/data) or upload a CSV / Excel file.
</Note>

## List files

```http
GET /v1/files?user=alice
Authorization: Bearer mb_live_*
```

```json
{
  "object": "list",
  "data": [
    { "file_key": "uploads/20260616_xxx.pdf", "filename": "checkup_2026.pdf", "file_type": "application/pdf", "created_at": "2026-06-16T07:40:00Z" }
  ],
  "subject": "alice"
}
```

The file list is wrapped in an object: `data` holds the array of files, and `subject` echoes the Subject the list belongs to.

## Fetch parsed text

```http
GET /v1/files/{file_key}
Authorization: Bearer mb_live_*
```

Returns the parsed text and abstract for one file — the same content the agent's `read_file` tool sees.

## Delete a file

```http
DELETE /v1/files/{file_key}
Authorization: Bearer mb_live_*
```

Deletes one uploaded file (and its parsed text) immediately.

```bash
curl -X DELETE "https://mcp.thetahealth.cn/v1/files/uploads/20260616_xxx.pdf" \
  -H "Authorization: Bearer $MIROBODY_API_KEY"
```

## Erase a Subject

```http
DELETE /v1/subjects/{user}
Authorization: Bearer mb_live_*
```

The right-to-be-forgotten sweep: erases **all** of a user's data — every structured record *and* every uploaded file. Use it to honor an end-user deletion request.

```bash
curl -X DELETE "https://mcp.thetahealth.cn/v1/subjects/alice" \
  -H "Authorization: Bearer $MIROBODY_API_KEY"
```

<Note>
These deletes require a key with the `data:delete` (or `data:write`) scope. To delete structured records only (without touching files), see [`DELETE /v1/data`](/en/api-reference/data#delete-records-right-to-be-forgotten).
</Note>
