---
title: "Data"
description: "POST /v1/data, GET /v1/data — write and read structured health records."
---

Write structured records straight into a Subject's store, and read them back. The chat agent reads this same data with its built-in tools.

## Write records

```http
POST /v1/data
Authorization: Bearer mb_live_*
Content-Type:  application/json
```

| Field | Type | Description |
| --- | --- | --- |
| `records` | array | Each record `{indicator, value, unit, time}`. Inserted in batches (1000 per batch). |
| `user` | string | Subject the records belong to (tenant-isolation key). |
| `retention` | string | `permanent` (default) / `1h` / `2h` / `6h` / `1d` / `session`. The hour grains auto-delete that many hours after write (still hard-capped at ≤ 24h). **`retention=none` is rejected here with HTTP `400`** — writing to the store but asking not to store is contradictory. For use-and-forget data, attach it as `health_context` with `retention=none` on [`POST /v1/chat/completions`](/en/api-reference/chat#inline-health-context) instead. |
| `session_id` | string | Required when `retention=session`. |

```bash
curl https://mcp.thetahealth.cn/v1/data \
  -H "Authorization: Bearer $MIROBODY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
        "user": "alice",
        "records": [
          {"indicator": "fasting_glucose", "value": 5.4, "unit": "mmol/L", "time": "2026-06-16T07:30:00Z"},
          {"indicator": "fasting_glucose", "value": 5.1, "unit": "mmol/L", "time": "2026-06-17T07:25:00Z"}
        ]
      }'
```

Response:

```json
{ "ingested": 2, "subject": "alice", "retention": "permanent" }
```

## Read records

```http
GET /v1/data?indicator=fasting_glucose&limit=100
Authorization: Bearer mb_live_*
```

| Query | Description |
| --- | --- |
| `user` | Subject to read from. |
| `indicator` | Optional filter by indicator name. |
| `limit` | `1`–`1000`. |
| `offset` | Pagination offset (default `0`). Skip this many records before returning `limit`. |

Expired records are purged before the read, so you always get the Subject's currently-valid data.

```python Python
import requests

r = requests.get(
    "https://mcp.thetahealth.cn/v1/data",
    headers={"Authorization": f"Bearer {MIROBODY_API_KEY}"},
    params={"user": "alice", "indicator": "fasting_glucose", "limit": 100, "offset": 0},
)
print(r.json())
```

The response wraps the records with pagination metadata:

```json
{
  "records": [ /* up to `limit` records */ ],
  "total": 248,
  "limit": 100,
  "offset": 0,
  "has_more": true
}
```

| Field | Description |
| --- | --- |
| `total` | Total matching records for the Subject (and `indicator`, if filtered). |
| `limit` | The `limit` that was applied. |
| `offset` | The `offset` that was applied. |
| `has_more` | `true` when `offset + limit < total` — fetch the next page by advancing `offset`. |

## Delete records (right to be forgotten)

```http
DELETE /v1/data?user=alice&indicator=fasting_glucose
Authorization: Bearer mb_live_*
```

Deletes the Subject's structured records. `indicator` is optional — omit it to delete **all** of the Subject's records.

| Query | Description |
| --- | --- |
| `user` | Subject whose records to delete. |
| `indicator` | Optional — restrict the delete to a single indicator. Omit to delete every record for the Subject. |

```bash
# delete just one indicator
curl -X DELETE "https://mcp.thetahealth.cn/v1/data?user=alice&indicator=fasting_glucose" \
  -H "Authorization: Bearer $MIROBODY_API_KEY"

# delete all of the Subject's structured records
curl -X DELETE "https://mcp.thetahealth.cn/v1/data?user=alice" \
  -H "Authorization: Bearer $MIROBODY_API_KEY"
```

To erase **everything** for a user — all structured records *and* uploaded files — use [`DELETE /v1/subjects/{user}`](/en/api-reference/files#erase-a-subject).

<Note>
Deletes require a key with the `data:delete` (or `data:write`) scope.
</Note>

<Note>
For one-off data that should not persist, attach it inline to a chat request as `health_context` with `retention="none"` instead of writing it here — see [Chat → Inline data](/en/api-reference/chat#inline-health-context).
</Note>
