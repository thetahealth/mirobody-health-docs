---
title: "Data"
description: "POST /v1/data, GET /v1/data — write and read structured health records."
---

Write structured records straight into a Subject's store, and read them back. The chat agent reads this same data with its built-in tools.

## Write records

```http
POST /v1/data
Authorization: Bearer mb_live_*
Content-Type:  application/json
```

| Field | Type | Description |
| --- | --- | --- |
| `records` | array | Each record `{indicator, value, unit, time}`. Inserted in batches (1000 per batch). |
| `user` | string | Subject the records belong to (tenant-isolation key). |
| `retention` | string | **Required — there is no default.** One of `permanent` / `1h` / `2h` / `6h` / `1d` / `session` (`persistent` is accepted as a synonym for `permanent`). The hour grains auto-delete that many hours after write (still hard-capped at ≤ 24h). Omitting it returns HTTP `400` (`code: invalid_retention`). **`retention=none` is also rejected here with `400`** — writing to the store but asking not to store is contradictory; for use-and-forget data attach it inline as `health_context` with `retention=none` on [`POST /v1/chat/completions`](/en/api-reference/chat#inline-health_context) instead. Only a *missing* value is rejected — unknown or out-of-range values are currently accepted without validation. |
| `session_id` | string | Scopes `retention=session` records. **Not currently enforced** — accepted without it, but records written without a `session_id` can't be targeted by a session delete. |

```bash
curl https://test-mirobody-api.thetahealth.ai/v1/data \
  -H "Authorization: Bearer $MIROBODY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
        "user": "alice",
        "retention": "permanent",
        "records": [
          {"indicator": "fasting_glucose", "value": 5.4, "unit": "mmol/L", "time": "2026-06-16T07:30:00Z"},
          {"indicator": "fasting_glucose", "value": 5.1, "unit": "mmol/L", "time": "2026-06-17T07:25:00Z"}
        ]
      }'
```

Response:

```json
{ "status": "ok", "ingested": 2, "subject": "alice" }
```

## Read records

```http
GET /v1/data?indicator=fasting_glucose&limit=100
Authorization: Bearer mb_live_*
```

| Query | Description |
| --- | --- |
| `user` | Subject to read from. |
| `indicator` | Optional filter by indicator name. |
| `limit` | `1`–`1000`. |
| `offset` | Pagination offset (default `0`). Skip this many records before returning `limit`. |

Expired records are purged before the read, so you always get the Subject's currently-valid data.

```python Python
import requests

r = requests.get(
    "https://test-mirobody-api.thetahealth.ai/v1/data",
    headers={"Authorization": f"Bearer {MIROBODY_API_KEY}"},
    params={"user": "alice", "indicator": "fasting_glucose", "limit": 100, "offset": 0},
)
print(r.json())
```

The response is an OpenAI-style list. Records are under **`data`**, and each record's `value` is a **formatted string** (e.g. `"5.4 mmol/L"`), not a number:

```json
{
  "object": "list",
  "data": [
    {
      "indicator": "fasting_glucose",
      "value": "5.4 mmol/L",
      "parsed_value": null,
      "parsed_unit": null,
      "time": "2026-06-16T07:30:00+00",
      "source": "api",
      "comment": ""
    }
  ],
  "has_more": false,
  "subject": "alice"
}
```

| Field | Description |
| --- | --- |
| `data` | The array of records (up to `limit`). |
| `has_more` | `true` when more records remain past this page — advance `offset` to fetch them. |
| `subject` | The Subject the records belong to (resolved from `user`). |

<Note>
The response does **not** currently return `total`, `limit`, or `offset` fields — page by advancing the `offset` query param and reading until `has_more` is `false`.
</Note>

## Deleting data

<Note>
Explicit delete endpoints are **not yet available** in `/v1` — `DELETE /v1/data`, `DELETE /v1/files/{file_key}`, and `DELETE /v1/subjects/{user}` are not implemented on the current cluster. Today, records are removed by **[retention](/en/api-reference/overview#data-retention) expiry**: the time-bounded grains (`1h`–`1d`) auto-delete, and `session` data is capped at 24h. For a full data-erasure request, contact [Mirobody Support](mailto:developer@thetahealth.ai) — see [Compliance](/en/api-reference/compliance).
</Note>

<Note>
For one-off data that should not persist at all, attach it inline to a chat request as `health_context` with `retention="none"` instead of writing it here — see [Chat → Inline data](/en/api-reference/chat#inline-health_context).
</Note>
