<Warning>
**Shipping soon — not yet available.** MCP tool attachment is **not implemented on the current clusters**; requests carrying MCP configuration are not honored. It will land on the [Agent API](/en/api-reference/responses) using the OpenAI-standard `tools: [{"type": "mcp"}]` shape. This page describes the roadmap, not current behavior. Today, bring your own tools via [client function tools](/en/api-reference/function-calling) instead.
</Warning>

## What's coming

The chat agent ships with built-in tools over the Subject's health data ([Function calling → Built-in server tools](/en/api-reference/function-calling#built-in-server-tools)). MCP attachment will let you extend it with **your own remote MCP servers**: the platform connects, lists your server's tools, and lets the agent call them alongside the built-ins during a turn.

<Note>
This is **not** "Mirobody as an MCP server." It's the reverse: you bring external MCP tools *into* the agent's data-gathering. The platform proxies the calls.
</Note>

## Planned shape (subject to change)

On the **Agent API**, the OpenAI Responses `tools` shape:

```jsonc
// POST /v1/responses — ROADMAP, not yet live
{
  "model": "mirobody-expert",
  "input": "Cross-check my labs against our formulary.",
  "user": "alice",
  "tools": [
    {
      "type": "mcp",
      "server_label": "formulary",
      "server_url": "https://tools.example.com/mcp",
      "authorization": "Bearer ..."
    }
  ]
}
```

Your MCP tools' calls will surface through the same trace channels as other tools; security posture (allowed-tool filtering, approval flow) will be documented at launch.

## Today's alternative

Everything MCP attachment enables is achievable now with [**client function tools**](/en/api-reference/function-calling): declare the tools, execute them in your process (including by proxying to your own MCP server), and continue the run with `function_call_output` — the openai-agents SDK automates the whole loop.
