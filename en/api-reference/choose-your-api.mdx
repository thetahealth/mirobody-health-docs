title: "Choose Your API"
description: "Answers API vs Agent API — pick the right surface in one minute."
---

Mirobody exposes the same grounded health agent through **two surfaces**, both answering from the Subject's real, standardized health data with a traceable tool trace.

<CardGroup cols={2}>
  <Card title="Answers API" icon="comment-dots" href="/en/api-reference/chat">
    `POST /v1/chat/completions` — closed, one-shot, evidence-backed completions. No client tools, no state. Drop-in OpenAI swap.
  </Card>
  <Card title="Agent API" icon="robot" href="/en/api-reference/responses">
    `POST /v1/responses` — OpenAI Responses-compatible. Multi-turn state, your own function tools, `response.*` streaming.
  </Card>
</CardGroup>

Answers speaks the older Chat Completions protocol and stays deliberately closed — no client tools, no stored state, just `chat.completion.chunk` streaming — which is what makes it trivial to drop into an existing OpenAI integration or [wrap as a single tool](/en/api-reference/use-as-a-tool) inside a larger agent. Agent speaks the newer Responses protocol: it keeps state between turns (`store`, `previous_response_id`, `session_id`), accepts your own function tools with a full `function_call` handoff, streams `response.*` events, and is what the `openai-agents` SDK talks to out of the box — change only the `base_url`.

Everything else is identical either way: the same server-side grounding over your [standardized data](/en/api-reference/data), the same `mirobody-flash` / `mirobody-expert` models, the same Subject-based tenancy, the same usage accounting.
