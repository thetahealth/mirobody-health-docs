Mirobody exposes the same grounded health agent through **two surfaces**, both answering from the Subject's real, standardized health data with a traceable tool trace.

<CardGroup cols={2}>
  <Card title="Answers API" icon="comment-dots" href="/en/api-reference/chat">
    `POST /v1/chat/completions` — closed, one-shot, evidence-backed completions. No client tools, no state. Drop-in OpenAI swap.
  </Card>
  <Card title="Agent API" icon="robot" href="/en/api-reference/responses">
    `POST /v1/responses` — OpenAI Responses-compatible. Multi-turn state, your own function tools, `response.*` streaming.
  </Card>
</CardGroup>

## What's different

| | Answers (`/v1/chat/completions`) | Agent (`/v1/responses`) |
| --- | --- | --- |
| Protocol | OpenAI Chat Completions | OpenAI Responses |
| State | Stateless — resend history | `store`, `previous_response_id`, `session_id` |
| Client tools | Rejected (`400 unsupported_parameter`) | `tools` with `function_call` handoff |
| Streaming | `chat.completion.chunk` frames | `response.*` event protocol |
| Server tool trace | `message.tool_steps` + extensions | top-level `tool_steps` + `response.mirobody_tool_call` |

## What's identical

Grounding ([Data Plane](/en/api-reference/data)), models (`mirobody-flash` / `mirobody-expert`), Subject tenancy via `user`, and usage accounting are the same on both.
