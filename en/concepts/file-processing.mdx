## Overview

Mirobody can ingest health documents — lab reports, spreadsheets, scanned records — and turn them into content a chat turn can carry. The work is done by the **document transcoder** in `src/transcode/` ([`document.hpp`](https://github.com/thetahealth/mirobody/blob/main-v2/src/transcode/document.hpp) / `document.cpp`), the document counterpart to the image transcoder. Its strategy is **text first, image (and OCR) fallback**.

<Info>
The transcoder decomposes a document into text and image parts; a vision-capable LLM then reads them during the chat turn.
</Info>

## Supported formats

The container format is sniffed from the leading magic bytes, then routed to a handler:

| Format | Library | Build gate | Output |
|--------|---------|-----------|--------|
| **PDF** | PDFium | `MIROBODY_ENABLE_PDF` (default **off**) | Text per page; scanned pages → image (+ OCR) |
| **.xlsx** | xlnt | on when xlnt is found (always on Windows via vcpkg) | One Markdown table per worksheet |
| **.xls** (legacy BIFF) | vendored libxls | `MIROBODY_ENABLE_XLS` (default **off**) | One Markdown table per sheet |
| **.csv** | built in (RFC 4180) | always available | A single Markdown table |

With a gate off, the input still sniffs but `process()` throws `DocumentError("... not built")` — nothing is silently mis-handled.

<Note>
Images uploaded on their own are handled by the sibling **image transcoder** (`src/transcode/image.hpp`), which fits an image to a target vision model's input limits. Audio and genetic-file handling from the old docs are **not** part of this C++ transcoder.
</Note>

## The pipeline

<Steps>
  <Step title="Sniff the format">
    `detect_format()` reads the magic bytes: `%PDF-` → PDF, `PK\x03\x04` → xlsx (any OOXML zip), the OLE2 signature → legacy .xls, anything else non-empty → CSV.
  </Step>

  <Step title="PDF — text layer first">
    For each page, the embedded **text layer is extracted** as a text part. A page whose extractable text is below a small threshold (a scan) is treated as an image instead.
  </Step>

  <Step title="Scanned page — rasterize → vision-compliant image">
    A scanned page is rasterized (default 150 dpi), repacked to PNG, and run through `image::Transcoder` so it satisfies the target vision model's limits (Qwen-VL by default; Gemini / GPT limits are also available). It is emitted as an **image part** the chat turn inlines for the vision model.
  </Step>

  <Step title="OCR (optional)">
    In a `MIROBODY_ENABLE_OCR` build (which additionally requires `MIROBODY_ENABLE_PDF`), the same raster is also run through **Tesseract** and the recovered text appended as a text part — so a scan yields both the image and its OCR text. OCR language defaults to `eng` (e.g. `eng+chi_sim` for mixed docs).
  </Step>

  <Step title="Spreadsheets — always text">
    Each worksheet (xlsx / xls) or the whole CSV is rendered to a **GitHub-flavored Markdown table**; spreadsheet handling never produces image parts.
  </Step>
</Steps>

The result is a `Document`: an ordered list of `Part`s, each either **Text** (UTF-8 / Markdown) or **Image** (an already vision-compliant image), tagged with the 1-based source page / sheet. A `max_pages` cap can bound very large documents; when it trips, a trailing note is appended so nothing is silently dropped. `Transcoder::to_markdown()` renders the whole document to one Markdown blob for callers (and the CLI) that want a single string.

## Units get normalized to UCUM

Values pulled out of documents and charts are normalized by the terminology engine in `src/fhir/units/`: a free-text "value + unit" string becomes a canonical **UCUM** unit plus the LOINC PROPERTY family that disambiguates it. This is pure local computation — no DB, no embedding API.

```
"MG/DL"              → "mg/dL"
"毫摩尔每升"          → "mmol/L"
"mmHg"               → "mm[Hg]"
"<5.6 mg/dL"         → value 5.6, unit "mg/dL", comparator "<"
```

Multilingual input is supported (en, zh, ja, ko, ru, de, fr, es).

## What happens to the data

The transcoded parts are read by the agent during a chat turn: uploaded files are stored per-user and the agent lists / reads them through the `list_files` and `read_file` MCP tools, feeding text parts and vision-compliant image parts to the LLM.

<Warning>
**At least one LLM key (or on-device Gemma) is required** to make sense of documents — uploading and storing a file works without one, but extracting meaning from PDFs / images needs a model. Text-only formats (CSV / xlsx) render to Markdown with no model.
</Warning>

<Note>
Writing extracted lab values back into the FHIR store as coded `Observation`s (the document → indicator → FHIR pipeline) is **in progress**: UCUM unit normalization and the RESTful FHIR store are shipped; terminology resolve (SNOMED CT / LOINC / RxNorm) is still landing and automatic write-back is planned.
</Note>
