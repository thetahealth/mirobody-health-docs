## Overview

Health data enters Mirobody through three doors, lands in one **embedded FHIR R4 store**, and is read back by the agent through **MCP tools** during a chat turn. Everything runs inside the single C++ core вҖ” standalone on a server, or on-device inside the mobile apps.

<Info>
This is not the old Python `StandardPulseData` вҶ’ `th_series_data` pipeline. There is no `pull_from_vendor_api` / `format_data` provider method chain; the current model is: sources write FHIR resources into the embedded store, and the agent reads them via tools.
</Info>

## Door 1 вҖ” Vendors (server-to-server)

For sources with a server API (wearable brands, B2B aggregators, EHRs), a [`vendor::Vendor`](/en/concepts/providers) client pulls data on the user's behalf. The path is ownership-gated:

<Steps>
  <Step title="Bind">
    The user records their external account id: `POST /vendors/{id}/bind` вҶ’ a PENDING link in `user_vendor_accounts`.
  </Step>
  <Step title="Verify">
    Ownership is proven through the vendor's consent flow: `POST /vendors/{id}/bind/verify` вҶ’ the link is marked VERIFIED. A `UNIQUE(vendor_id, external_user_id)` constraint stops two users claiming one account.
  </Step>
  <Step title="Fetch">
    `GET /vendors/{id}/data?domain=&start=&end=` resolves the verified link, builds the vendor client from config, and calls `Vendor::fetch(user_id, domain, start_iso, end_iso)`, returning the source's JSON.
  </Step>
</Steps>

FHIR-native vendors (the generic `ehr` SMART-on-FHIR client, plus Vitalera / HealthConnect / Redox / Particle Health / Metriport-Medical) already return FHIR R4 resources; the rest return each vendor's own JSON. The SMART-on-FHIR EHR connect flow (`/health/ehr/authorize` вҶ’ `/health/ehr/callback` вҶ’ `/health/ehr/sync`) fetches `Observation`s and persists each one through the FHIR store.

## Door 2 вҖ” On-device stores (POST FHIR Observations)

Apple Health, Samsung Health, Google Health Connect, and Xiaomi expose **no server API**, so they have no vendor client. Instead the **host app reads samples on-device** through the platform SDK (Apple HealthKit on iOS; Health Connect + HMS Health Kit on Android), maps them to FHIR `Observation`s, and **writes them to the embedded FHIR R4 endpoint** вҖ” the one door a web page alone cannot open, which is why the native apps exist. The FHIR write model deliberately mirrors Android Health Connect's medical-records rules.

## Door 3 вҖ” File uploads

Uploaded documents (lab reports, spreadsheets, scans) are decomposed by the document transcoder into LLM-consumable parts, then the LLM reads them during chat. Uploads are stored per-user; the agent lists and reads them with the `list_files` / `read_file` MCP tools. See [File Processing](/en/concepts/file-processing).

<Note>
The document вҶ’ structured-indicator вҶ’ FHIR pipeline (parsing a report into coded lab values written back as FHIR) is **in progress** вҖ” unit normalization to UCUM and the RESTful FHIR store are shipped; full terminology mapping and automatic write-back are planned. *(verify against `src/fhir/README.md` phases.)*
</Note>

## The store вҖ” embedded FHIR R4

All three doors converge on the embedded FHIR R4 endpoint in `src/fhir/`:

- Resources are stored as generic validated JSON in one portable `fhir_resources(user_id, resource_type, resource_id, version_id, updated_at, deleted_at, content)` table, across whichever SQL backend was compiled in (SQLite on mobile, PostgreSQL on desktop, вҖҰ).
- Each user has an **independent resource space** (single-patient model); every FHIR route requires a bearer JWT.
- Units are normalized to **UCUM** (`mg/dL`, `mmol/L`, `mm[Hg]`, вҖҰ) by the terminology engine in `src/fhir/units/`.
- Deletes are soft (a later read returns `410 Gone`).

Routes: `GET /fhir/metadata`, `POST /fhir` (batch Bundle), and `GET/POST/PUT/DELETE /fhir/{type}[/{id}]`. Bodies are `application/fhir+json`; errors come back as an `OperationOutcome`.

## Reading it back вҖ” the agent + MCP tools

When a user chats, the single agent (`src/chat/` + `res/agents/base.cpp`) picks an LLM client by provider, builds a system prompt, and streams the turn back вҖ” calling **MCP tools** as needed. The health-relevant tools:

- **`family_health`** вҖ” read-only FHIR data for a care-circle member, gated by the sharing level they granted.
- **`list_files`** / **`read_file`** вҖ” enumerate and read the user's uploaded documents.
- **`whoami`** вҖ” the current user / subject.
- **`render_chart`** вҖ” turn data into a chart.

The agent reads from the same per-user FHIR store the three doors wrote to, so device data, EHR records, and uploaded documents all appear as one unified history regardless of source.
