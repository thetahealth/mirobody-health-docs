## What a vendor is

In Mirobody a **vendor** is a C++ client for one health-data source, living one-per-file under `src/health/vendor/`. Every vendor implements the same abstract interface — `vendor::Vendor` in [`src/health/vendor/vendor.hpp`](https://github.com/thetahealth/mirobody/blob/main-v2/src/health/vendor/vendor.hpp) — so the rest of the engine talks to a wearable brand, a B2B aggregator, and a hospital EHR through one contract, and picks the concrete client at runtime by **id**.

<Info>
Each data source is a **`vendor::Vendor`**, compiled in and resolved by id through a registry.
</Info>

## The four buckets

The per-vendor clients are sorted into four directories under `src/health/vendor/`:

  <Card title="Clinical EHRs" icon="hospital">

    `phone/` — smartphone-vendor stores that expose a cloud API. Today: **Huawei** Health Kit

<Note>
**On-device-only stores have no vendor client.** Apple Health, Samsung Health, Google Health Connect, and Xiaomi / Mi Fitness expose no server-callable API, so they are **not** `vendor::Vendor`s. Instead the host app reads samples on-device and POSTs them as FHIR `Observation`s to the embedded FHIR endpoint. See [Data Flow](/en/concepts/data-flow).
</Note>

Device *brands* with their own OAuth REST APIs can be reached two ways: **through an aggregator** (Terra, Validic, Thryve, Rook, Spike), or via a dedicated **first-party client**. Seven brands ship a direct client — **Fitbit, Withings, Garmin, Dexcom, Oura, Whoop, Polar** — each with a real OAuth flow and `fetch` against the brand's public API, for deployments that want the brand's data without an aggregator. Brands without a client stay aggregator-only unless someone adds one.

## The vendor contract

Every vendor derives from `VendorBase` and overrides the operations whose wire contract is confirmable from the source's public docs. The interface (abbreviated from `vendor.hpp`):

```cpp
namespace mirobody { namespace vendor {

class Vendor {
public:
  virtual ~Vendor() = default;

  // Static metadata (no network, no credentials).
  virtual const VendorInfo& info() const = 0;

  // Begin consumer-mediated consent; returns the entry point handed to the user
  // (usually an OAuth authorize URL).
  virtual std::string authorize_url(const std::string& redirect_uri,
                                    const std::string& state,
                                    const std::string& user_id,
                                    const std::string& provider) = 0;

  // The data sources a connected user can link, as the vendor's JSON.
  virtual std::string list_providers(const std::string& user_id) = 0;

  // Fetch `domain` for `user_id` over [start_iso, end_iso], as the vendor's JSON.
  virtual std::string fetch(const std::string& user_id,
                            DataDomain domain,
                            const std::string& start_iso,
                            const std::string& end_iso) = 0;

  // Validate and parse an inbound webhook delivery; returns the event as JSON.
  virtual std::string handle_webhook(const std::string& raw_headers,
                                     const std::string& body) = 0;

  // Revoke a user's authorization / disconnect them.
  virtual void revoke(const std::string& user_id,
                      const std::string& provider) = 0;

  // Exchange an OAuth2 authorization code (from authorize_url's redirect)
  // for a TokenSet at the vendor's token endpoint.
  virtual TokenSet exchange_code(const std::string& code,
                                 const std::string& redirect_uri) = 0;

  // Exchange a refresh_token for a fresh TokenSet.
  virtual TokenSet refresh(const std::string& refresh_token) = 0;
};

}}
```

- **`DataDomain`** is the normalized data category: `Activity`, `Sleep`, `HeartRate`, `Glucose`, `Nutrition`, `BodyMetrics`, `Labs`, `Clinical`.
- **`TokenSet`** carries `access_token` / `refresh_token` / `expires_in` from an OAuth2 token-endpoint call. The OAuth device brands implement `exchange_code` / `refresh` via a shared helper, `oauth2_token_request(...)` in [`src/health/vendor/oauth2.hpp`](https://github.com/thetahealth/mirobody/blob/main-v2/src/health/vendor/oauth2.hpp) (Withings uses its own envelope; Polar's tokens don't expire, so its `refresh` throws).
- **Time bounds** are ISO-8601 strings; empty means "the vendor's default window".
- **Data-returning operations hand back the vendor's own JSON** as a string. (A normalized model over that JSON is a later step; several FHIR-based vendors already return FHIR resources.)

### Honest stubs

`VendorBase` implements every operation as a stub that throws `VendorError("... not implemented")`. A concrete vendor overrides only the operations it can implement faithfully; the rest stay stubs **on purpose**, for one of three documented reasons:

- **gated / undocumented** — the contract (e.g. a webhook signature scheme) isn't public, so implementing it would mean fabricating an unverifiable scheme;
- **contract mismatch** — the documented endpoint needs an argument the interface doesn't carry;
- **no such endpoint** — the platform simply exposes no operation of that shape.

Each `<id>.cpp` header records why anything is left a stub. Fabricating a contract is treated as worse than an honest "not implemented".

## Metadata without credentials

Each vendor carries a `VendorInfo` describing its positioning, region, compliance posture, data domains, and integration style — queryable **without any credentials or network call**. This lets the registry describe the whole field for selection and comparison.

```cpp
struct VendorInfo {
  std::string id;              // stable lowercase key, e.g. "terra"
  std::string display_name;    // "Terra API"
  std::string positioning;
  std::string data_source_coverage;
  std::string docs_url;
  Region region;               // Global | US | EU | SelfHosted
  bool open_source;
  std::vector<std::string> compliance;   // "HIPAA", "GDPR", "SOC2_TYPE_II", …
  std::vector<DataDomain>   domains;
  std::vector<Integration>  integrations; // Rest | Sdk | Webhook | WebSocket | Fhir | Hl7 | WhiteLabel
  // …
};
```

## Resolved by id through a registry

The concrete vendor classes are never named outside their own translation unit. Callers reach them only through the registry ([`src/health/vendor/registry.hpp`](https://github.com/thetahealth/mirobody/blob/main-v2/src/health/vendor/registry.hpp)):

```cpp
namespace mirobody { namespace vendor {

// Every registered vendor id, in matrix order.
std::vector<std::string> vendor_ids();

// Static metadata for all vendors — no credentials, no network.
std::vector<VendorInfo> all_vendor_info();

// Construct the vendor for `id`; throws VendorError if unknown.
std::unique_ptr<Vendor> open_vendor(const std::string& id, const VendorConfig& config);

}}
```

### Configuration

A vendor is configured with a `VendorConfig` the server fills from the central config, using clean per-vendor keys — settable in `config.yml` or as the same-named environment variables (see `config.example.yml`):

```
<ID>_API_KEY        # bearer / API key (most REST vendors)
<ID>_CLIENT_ID      # OAuth client id
<ID>_CLIENT_SECRET  # OAuth client secret
<ID>_BASE_URL       # API host override; empty => vendor default
```

`<ID>` is the vendor id upper-cased (e.g. `HUMAN_API`, `FITBIT`, `OURA`). For vendors whose host is per-deployment (self-hosted or contract-gated, like the generic `ehr` client), `base_url` is **required** — the client refuses rather than guess a host. (The older `MIROBODY_VENDOR_<ID>_*` env convention is legacy; only the standalone `vendor` CLI still reads it on its own.)

## From HTTP to a vendor

The health module (`src/health/`) is the HTTP front end that sits on top of these clients. Access is **ownership-gated**: a user must prove they own the external account before any data is read.

| Method | Route | Purpose |
|--------|-------|---------|
| POST | `/vendors/{id}/bind` | Record the user's external account id as a PENDING link |
| POST | `/vendors/{id}/bind/verify` | Prove ownership via the vendor's consent flow → VERIFIED |
| GET | `/vendors/{id}/data?domain=&start=&end=` | Fetch data for a verified link over a time range (`Vendor::fetch`) |

The SMART-on-FHIR EHR connect flow is browser-driven and has its own routes under `/health/ehr/*` (`providers`, `authorize`, `callback`, `sync`). Every route requires a bearer JWT and is scoped to the authenticated user.

<Warning>
`/api/providers` is a **different** endpoint — it lists the chat **LLM providers** (OpenAI, Gemini, …) for the model picker, not health vendors. Health-data sources are the `vendor::Vendor` clients described here.
</Warning>
