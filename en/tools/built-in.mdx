## Overview

Mirobody ships **9 built-in MCP tools**. Each is a small C++ file in [`res/mcp_tools/`](https://github.com/thetahealth/mirobody/tree/main-v2/res/mcp_tools) that self-registers at compile time (see [Adding Custom Tools](/en/tools/adding-tools)). They're exposed over the [`/mcp` endpoint](/en/tools/mcp-integration) for external clients and are also handed to the built-in agent as function-call descriptors, so the OpenAI and Gemini clients can invoke them locally in-process.

<Info>
Tools marked **auth required** only run for an authenticated caller — the engine resolves the caller's identity (JWT or personal MCP secret) before dispatch and injects it as `UserInfo`. Tools without auth (like `echo` and `render_chart`) run for anyone.
</Info>

| Tool | Auth | What it does |
|------|:----:|--------------|
| [`list_files`](#list_files) | ✅ | List the files the user has uploaded |
| [`read_file`](#read_file) | ✅ | Read one uploaded file by its `file_key` |
| [`family_health`](#family_health) | ✅ | Read recent FHIR `Observation`s for the user or a shared care-circle member |
| [`whoami`](#whoami) | ✅ | Confirm authentication and return the session id |
| [`recall_memory`](#recall_memory) | ✅ | Retrieve the user's most relevant long-term memories |
| [`remember`](#remember) | ✅ | Store a durable fact in long-term memory |
| [`render_chart`](#render_chart) | — | Draw a chart in the chat UI (Apache ECharts) |
| [`summarize_conversation`](#summarize_conversation) | ✅ | Set a short title for the current conversation |
| [`echo`](#echo) | — | Echo text back (example / connectivity test) |

## File tools

### list_files

Lists the files the authenticated user has uploaded. Each entry carries the file's name, MIME type, `file_key`, and upload time. When text was extracted from the file at upload (images, PDFs, scans), the entry's MIME is reported as `text/plain` and `read_file` will return that extracted text as `content` — even though the original isn't a text format.

<ParamField body="(no parameters)">
  The caller is resolved from auth; no arguments are required.
</ParamField>

Returns a JSON array of `{ filename, file_key, mime, uploaded_at }`.

### read_file

Fetches one of the user's uploaded files by its `file_key` (as returned by `list_files`). Ownership is enforced through the per-user file index, so a caller can't read another user's objects by guessing keys. Text-ish files (and any file with extracted text) are returned inline as `content`; other binary types return a temporary signed `url` instead of inlining bytes the model can't use.

<ParamField body="file_key" type="string" required>
  The `file_key` of the file to read, as returned by `list_files`.
</ParamField>

Returns `{ filename, mime, content }` for text, or `{ filename, mime, url, note }` for binaries.

## Health tools

### family_health

Reads recent health records — FHIR `Observation` resources — for the user or a care-circle member who has shared their health data. This is how the agent answers questions like *"how did I sleep?"* or *"how is Mom doing?"*.

The `member` argument resolves to a target user: empty or `"me"` is the caller; a numeric id, name, nickname, or email matches someone who shared their data. The read is gated twice — the person must be in the caller's health-share list, and `can_read_health` re-authorizes the read. Only members who shared their data are reachable.

<ParamField body="member" type="string">
  Whose health to read: a care-circle member's name / nickname / email / id, or `"me"` (or omit) for the caller. When the chat composer's "currently for" picker has selected a member, that member is used by default.
</ParamField>

<ParamField body="count" type="integer">
  How many recent observations to return. Default `20`, max `100`.
</ParamField>

Returns `{ subject, observation_count, observations: [{ code, value, time }, …] }`.

## Memory tools

### recall_memory

Retrieves the user's most relevant long-term memories (stored facts, preferences, and past context) for a query, so the model can ground a turn in what it has learned about the user before. Requires long-term memory to be enabled on the server (otherwise returns a clean tool error).

<ParamField body="query" type="string" required>
  What to search the user's memories for.
</ParamField>

<ParamField body="top_k" type="integer">
  Maximum number of memories to return (default 5).
</ParamField>

Returns `{ memories: [{ id, text, kind, score, created_at }, …] }`.

### remember

Stores a durable fact about the user in long-term memory (a stable preference, goal, or personal detail worth recalling later), so a future turn can `recall_memory` it. The model decides what's worth keeping.

<ParamField body="text" type="string" required>
  The fact to remember, written as a self-contained sentence.
</ParamField>

<ParamField body="kind" type="string">
  Category tag: `fact` | `preference` | `episode` (default `fact`).
</ParamField>

Returns `{ id, stored }`.

## Chat / UI tools

### render_chart

Draws a chart directly in the user's chat. The model calls it with a complete [Apache ECharts](https://echarts.apache.org/) `option` object; the chat stream turns the call into a chart event the frontend renders with `echarts.setOption()`. The option travels to the client out-of-band via the call arguments, so the model's context isn't bloated by echoing it back — the tool itself just validates the option and returns a short acknowledgement.

<ParamField body="option" type="object" required>
  A complete Apache ECharts option object, e.g. `{"xAxis":{"type":"category","data":["Mon","Tue"]},"yAxis":{"type":"value"},"series":[{"type":"line","data":[70.1,70.4]}]}`.
</ParamField>

<ParamField body="title" type="string">
  Optional short caption shown alongside the chart.
</ParamField>

### summarize_conversation

Records a concise summary (a short title, ≤ 200 chars) of the current conversation, used as its label in the user's chat history. It overwrites the placeholder title the engine seeds from the opening question. The model writes the summary itself once it understands what the user wants.

<ParamField body="summary" type="string" required>
  A short, self-contained summary of the conversation (≤ 200 chars).
</ParamField>

Returns `{ updated: true }`.

## Example / test tools

### whoami

Confirms the caller is authenticated and returns their (opaque) session identifier. When the turn is "currently for" a care-circle member the caller may read, it also flags `acting_for_member`. It never echoes any internal user primary key back out.

<ParamField body="(no parameters)" />

Returns `{ authenticated: true, session_id, acting_for_member? }`.

### echo

Echoes the provided text back. It's the reference tool — the smallest possible example of the drop-a-file registration pattern (see [`echo.cpp`](https://github.com/thetahealth/mirobody/blob/main-v2/res/mcp_tools/echo.cpp)) — and doubles as a connectivity check.

<ParamField body="text" type="string" required>
  Text to echo back.
</ParamField>

Returns `{ echo: "<text>" }`.

## How a tool is defined

Every tool is one `res/mcp_tools/<name>.cpp` that declares a `Tool` — name, description, an `auth` flag, a small table of `Param`s, and a handler — then self-registers with `MIROBODY_REGISTER_TOOL(...)`. C++11 has no runtime reflection, so parameters are declared explicitly (they expand into the MCP `inputSchema` and the OpenAI / Gemini function descriptors) and checked at compile time. Here's the whole of `echo`:

```cpp res/mcp_tools/echo.cpp
#include "mcp/tool.hpp"
#include <rapidjson/document.h>

namespace {
using namespace mirobody::mcp;

Result echo(const Args& args, const UserInfo&, const ToolContext&) {
    const std::string text = args.str("text");
    rapidjson::Document d;
    d.SetObject();
    rapidjson::Document::AllocatorType& a = d.GetAllocator();
    d.AddMember("echo",
                rapidjson::Value(text.c_str(),
                                 static_cast<rapidjson::SizeType>(text.size()), a),
                a);
    return Result::ok(to_json(d));
}

const Tool kEcho = {
    "echo",
    "Echo back the provided text.",
    false,                                                    // auth
    { Param("text", Type::String, Required, "Text to echo back") },
    &echo,
};
}   // namespace

MIROBODY_REGISTER_TOOL(kEcho);
```
