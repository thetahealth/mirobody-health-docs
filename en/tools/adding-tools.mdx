## Overview

Tools in Mirobody are **C++ files in [`res/mcp_tools/`](https://github.com/thetahealth/mirobody/tree/main/res/mcp_tools)**. Each file declares a `Tool` and self-registers at compile time; `CMakeLists.txt` globs every `.cpp` in that directory into the build, so the whole workflow is: **drop a new file, rebuild, and the tool is live** — no manual registration, no router wiring, no JSON schema by hand.

<Info>
This is a **compile-time** mechanism, not a runtime drop-in. There is no Python tool loader in the C++ engine (Mirobody) — parameters are declared in a small table and the registry expands them into the MCP `inputSchema` and the OpenAI / Gemini function descriptors.
</Info>

## Anatomy of a tool

A tool declares five things (the sixth, `raw_input_schema`, is an optional escape hatch):

| Field | Type | Purpose |
|-------|------|---------|
| `name` | `std::string` | The tool name the model calls. |
| `description` | `std::string` | What the tool does and when to use it — the model reads this verbatim. |
| `auth` | `bool` | `true` requires an authenticated caller; the engine injects `UserInfo`. |
| `params` | `std::vector<Param>` | Declared parameters (`name`, `Type`, `Required`/`Optional`, description). |
| `handler` | function | `Result(const Args&, const UserInfo&, const ToolContext&)`. |

Parameter types are `Type::String`, `Integer`, `Number`, `Boolean`, `Array`, `Object`. Read arguments off `Args` with typed accessors (`args.str("x")`, `args.integer("n", 20)`, `args.raw()` for arbitrary shapes). Return `Result::ok(json)` or `Result::error("message")`.

## Quick start

The reference tool is [`echo.cpp`](https://github.com/thetahealth/mirobody/blob/main/res/mcp_tools/echo.cpp) — the smallest complete example. Copy it and go.

<Steps>
  <Step title="Create the file">
    Add a new `.cpp` in `res/mcp_tools/`:

    ```bash
    touch res/mcp_tools/weather.cpp
    ```
  </Step>

  <Step title="Declare a Tool and a handler">
    Include `mcp/tool.hpp`, write the handler, declare the `Tool`, and register it:

    ```cpp res/mcp_tools/weather.cpp
    #include "mcp/tool.hpp"
    #include <rapidjson/document.h>

    namespace {
    using namespace mirobody::mcp;

    Result weather(const Args& args, const UserInfo&, const ToolContext&) {
        const std::string city = args.str("city");
        if (city.empty()) return Result::error("city is required");

        rapidjson::Document d;
        d.SetObject();
        rapidjson::Document::AllocatorType& a = d.GetAllocator();
        d.AddMember("city",
                    rapidjson::Value(city.c_str(),
                                     static_cast<rapidjson::SizeType>(city.size()), a), a);
        d.AddMember("temperature_c", 22, a);
        d.AddMember("condition", "Sunny", a);
        return Result::ok(to_json(d));
    }

    const Tool kWeather = {
        "weather",
        "Get the current weather for a city. Call this when the user asks about "
        "the weather or conditions in a named place.",
        false,                                                  // auth
        { Param("city", Type::String, Required, "City name, e.g. \"Boston\"") },
        &weather,
    };
    }   // namespace

    MIROBODY_REGISTER_TOOL(kWeather);
    ```

    <Check>
    The `MIROBODY_REGISTER_TOOL(...)` line at file scope is what wires the tool in — it initializes a file-scope static that calls `registry().add(...)`.
    </Check>
  </Step>

  <Step title="Rebuild">
    ```bash
    ./build.sh
    ```

    `res/mcp_tools/*.cpp` is globbed by CMake, so your new file is picked up with no other edits. Restart `./mirobody` and the tool is live.
  </Step>
</Steps>

## Tools that touch user data

Set `auth = true` and the engine resolves the caller (JWT or personal MCP secret) before dispatch, handing the identity in as `UserInfo`. This is the C++ analog of the old Python `user_info` parameter. Bail out on an unauthenticated call:

```cpp
Result my_tool(const Args& args, const UserInfo& user, const ToolContext& ctx) {
    if (!user.authed()) return Result::error("Not authenticated");
    // user.user_id is the caller's row id; scope everything to it.
    // ...
}
```

Anything a handler needs beyond its arguments and the caller identity comes through `ToolContext` — all **borrowed** pointers, any of which may be null if that backend isn't configured:

| Handle | Reaches | Used by |
|--------|---------|---------|
| `ctx.cache` | per-user indices (e.g. the uploaded-file index) | `list_files`, `read_file` |
| `ctx.storage` | object store for uploaded bytes / signed URLs | `read_file` |
| `ctx.db` | relational store | `family_health`, `summarize_conversation` |
| `ctx.memory` | long-term memory store | `recall_memory`, `remember` |

Tolerate a missing backend with a clean error, exactly as the built-ins do:

```cpp
if (ctx.memory == nullptr)
    return Result::error("Long-term memory is not enabled on this server");
```

<Tip>
For the read patterns — enforcing per-user ownership, returning FHIR observations, persisting to the relational store — read the built-ins alongside this page: [`list_files.cpp`](https://github.com/thetahealth/mirobody/blob/main/res/mcp_tools/list_files.cpp), [`read_file.cpp`](https://github.com/thetahealth/mirobody/blob/main/res/mcp_tools/read_file.cpp), [`family_health.cpp`](https://github.com/thetahealth/mirobody/blob/main/res/mcp_tools/family_health.cpp), [`remember.cpp`](https://github.com/thetahealth/mirobody/blob/main/res/mcp_tools/remember.cpp).
</Tip>

## Non-flat arguments

The declared `params` table generates a JSON Schema of flat scalars and arrays, which covers most tools. For a tool whose arguments aren't flat, read the raw object with `args.raw()` (as [`render_chart`](https://github.com/thetahealth/mirobody/blob/main/res/mcp_tools/render_chart.cpp) does with its ECharts `option`), and — if you need the advertised schema to match exactly — set the optional `raw_input_schema` field on the `Tool` to a schema string that's used verbatim.

## Best practices

<AccordionGroup>
  <Accordion title="Write the description for the model" icon="file-lines">
    The `description` and each `Param` description are the only context the model has when deciding whether and how to call your tool. Say **what it does and when to use it**, and give a concrete example value in the param description. This matters more than anything else for getting the tool called correctly.
  </Accordion>

  <Accordion title="Return structured JSON" icon="code">
    Build a small JSON object with rapidjson and return `Result::ok(to_json(d))`. Keep results compact — don't inline large blobs the model can't use (see how `read_file` returns a signed `url` for binaries instead of the bytes).
  </Accordion>

  <Accordion title="Fail cleanly" icon="triangle-exclamation">
    Return `Result::error("...")` for bad input, missing auth, or an unconfigured backend. The registry also catches any handler exception and turns it into an error, so a tool never crashes the server.
  </Accordion>

  <Accordion title="Scope to the caller" icon="lock">
    For `auth` tools, derive everything from `user.user_id`. Never use `subject_user_id` for writes or to expose other data — it's a read-only care-circle hint that only health-read tools should honor.
  </Accordion>
</AccordionGroup>

## Verify the tool loaded

Hit the [`/mcp` endpoint](/en/tools/mcp-integration) with `tools/list` and confirm your tool is in the returned array:

```bash
curl -X POST http://localhost:8080/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
```

If it's missing, check the build log for compile errors and confirm the file is under `res/mcp_tools/` and ends with a `MIROBODY_REGISTER_TOOL(...)` line.
