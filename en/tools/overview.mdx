## The agent

Mirobody runs **one simple agent** — "Base", defined in [`res/agents/base.cpp`](https://github.com/thetahealth/mirobody/blob/main/res/agents/base.cpp). It is deliberately minimal: there is no agent loop, no planner, no middleware. For each turn it:

1. **Picks an LLM client by provider name** (the requested `provider`, else the default).
2. **Builds a system prompt** — a fixed inline string with the current time, the response language, and guidance to fetch uploads (`list_files` / `read_file`) and health records (`family_health`).
3. **Streams the provider's events straight back**, letting the model call the built-in [MCP tools](/en/tools/built-in) as needed.

<Note>
You choose a **model** (`provider`), not an agent *type* — Mirobody runs a single agent.
</Note>

### Providers

The agent registers up to four provider clients (each keyed by the model name the `/api/providers` selector shows). The first three are always registered so they appear in the selector; one whose credentials aren't set simply errors at call time.

| Provider key | Model | Client | Tool execution |
|--------------|-------|--------|----------------|
| `gemini-2.5-flash` | Google Gemini 2.5 Flash (**default**) | `GeminiClient` | Local, in-process |
| `gpt-5-nano` | OpenAI GPT-5 nano | `OpenAIChatClient` | Local, in-process |
| `mirothinker-1.7` | MiroThinker (MiroMind) | `MiroThinkerClient` | Provider-native MCP via `MCP_PUBLIC_URL` |
| `gemma-4-e2b` | On-device Gemma 4 E2B | `OpenAIChatClient` (local endpoint) | Depends on serving stack |

The native apps also run **Gemma 4 E2B on-device** (LiteRT-LM on Android/iOS, llama.cpp on Electron) — offline, no key. `gemma-4-e2b` above is the same model served over a local OpenAI-compatible endpoint (Ollama / llama.cpp / vLLM), registered only when a base URL is configured.

<Info>
**How tools run differs by provider.** OpenAI and Gemini receive the tools as function-call descriptors and the engine executes each call **locally in-process** as the authenticated user. MiroThinker runs tools **provider-side** — MiroMind reaches back to your `/mcp` endpoint over the public internet at `MCP_PUBLIC_URL`.
</Info>

### Configuration

Providers are configured with flat keys in `config.yml` (see [Configuration](/en/configuration)). There is **no `PROVIDERS_DEEP`** block.

```yaml config.yml
# OpenAI (gpt-5-nano)
OPENAI_API_KEY: 'your_openai_key'
OPENAI_BASE_URL: 'https://api.openai.com'   # client appends /v1/chat/completions

# Google Gemini (gemini-2.5-flash, the default provider)
GOOGLE_API_KEY: 'your_google_key'

# MiroThinker (mirothinker-1.7) — provider-native MCP
MIROTHINKER_API_KEY: 'your_mirothinker_key'
MIROTHINKER_BASE_URL: ''      # optional override
MIROTHINKER_MODEL: ''         # optional override
MCP_PUBLIC_URL: 'https://your-domain.com/mcp'   # where MiroMind reaches your MCP

# On-device Gemma served over a local OpenAI-compatible endpoint (optional)
GEMMA_CHAT_BASE_URL: 'http://localhost:11434/v1'   # or OLLAMA_BASE_URL
GEMMA_CHAT_API_KEY: ''
GEMMA_CHAT_MODEL: 'gemma-4-e2b'
```

### What the agent covers

Personal-health questions almost always resolve to: *read the user's records → optionally read uploaded files → optionally recall long-term memory → answer, sometimes rendering a chart*. The built-in tools map directly onto that, and the model orchestrates them itself within a single streamed turn.

## Tools system

### What tools are

Tools are **C++ files in [`res/mcp_tools/`](https://github.com/thetahealth/mirobody/tree/main/res/mcp_tools)**. Each declares a `Tool` (name, description, an `auth` flag, a small `Param` table, and a handler) and self-registers at compile time. `CMakeLists.txt` globs every `.cpp` in that directory (with `CONFIGURE_DEPENDS`) into the build, so **dropping a new file and rebuilding is all it takes** — no manual registration, no hand-written JSON schema.

<Info>
This is a **compile-time** mechanism: tools are compiled in from `res/mcp_tools/`, not loaded from a runtime directory.
</Info>

### The nine built-in tools

| Tool | Auth | Purpose |
|------|:----:|---------|
| `list_files` | ✅ | List the user's uploaded files |
| `read_file` | ✅ | Read one uploaded file by `file_key` |
| `family_health` | ✅ | Read recent FHIR `Observation`s for the user or a shared care-circle member |
| `whoami` | ✅ | Confirm authentication, return the session id |
| `recall_memory` | ✅ | Retrieve the user's most relevant long-term memories |
| `remember` | ✅ | Store a durable fact in long-term memory |
| `render_chart` | — | Draw a chart in the chat UI (Apache ECharts) |
| `summarize_conversation` | ✅ | Set a short title for the current conversation |
| `echo` | — | Echo text back (example / connectivity test) |

See [Built-in Tools](/en/tools/built-in) for each tool's parameters and return shape.

### Adding a tool

Create `res/mcp_tools/<name>.cpp`, declare a `Tool`, and register it with the `MIROBODY_REGISTER_TOOL(...)` macro — then rebuild:

```cpp res/mcp_tools/echo.cpp
#include "mcp/tool.hpp"
#include <rapidjson/document.h>

namespace {
using namespace mirobody::mcp;

Result echo(const Args& args, const UserInfo&, const ToolContext&) {
    const std::string text = args.str("text");
    rapidjson::Document d;
    d.SetObject();
    rapidjson::Document::AllocatorType& a = d.GetAllocator();
    d.AddMember("echo",
                rapidjson::Value(text.c_str(),
                                 static_cast<rapidjson::SizeType>(text.size()), a), a);
    return Result::ok(to_json(d));
}

const Tool kEcho = {
    "echo",
    "Echo back the provided text.",
    false,                                                    // auth
    { Param("text", Type::String, Required, "Text to echo back") },
    &echo,
};
}   // namespace

MIROBODY_REGISTER_TOOL(kEcho);
```

Auth-scoped tools set `auth = true` and receive the caller's identity as `UserInfo`; anything else a handler needs (cache, object store, database, long-term memory) arrives through `ToolContext`. Full walkthrough: [Adding Custom Tools](/en/tools/adding-tools).

## The MCP endpoint

The engine exposes its tools over an MCP endpoint served by `src/mcp/`. It speaks **JSON-RPC 2.0**.

```
http://localhost:8080/mcp                 # authenticates via bearer JWT
http://localhost:8080/mcp/{secret}        # personal-MCP secret resolving to a user
```

Supported methods include `tools/list` and `tools/call`. For local development, tunnel the port (e.g. `ngrok http 8080`) and set `MCP_PUBLIC_URL` so remote clients — and MiroThinker's provider-native MCP — can reach it. See [MCP Integration](/en/tools/mcp-integration) to connect Claude, Cursor, and ChatGPT.

```bash
curl -X POST http://localhost:8080/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
```

## Where health data lives

Tools like `family_health` don't hit ad-hoc tables — they read through the **embedded FHIR R4 store** (`src/fhir/`), returning FHIR `Observation` resources. Data is normalized on the way in (units to UCUM), and the relational backend is chosen at **build time** via CMake (`-DMIROBODY_DATABASE_BACKEND=`): SQLite (mobile default), PostgreSQL (desktop default), MySQL, DuckDB, or ClickHouse. The SQL schemas live in [`res/sql/`](https://github.com/thetahealth/mirobody/tree/main/res/sql) (`pg`, `sqlite`, `mysql`, `duckdb`, `clickhouse`). See [Architecture](/en/concepts/architecture) for the full picture.
