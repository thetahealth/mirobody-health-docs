Mirobody is configured through a single **`config.yml`** file — a flat set of keys parsed by the C++ core. Copy the committed template to `config.yml` and edit that. The template documents defaults, optional integrations, and required production secrets; integrations remain disabled until you provide their credentials.

```bash
cp config.example.yml config.yml
```

<Info>
Configuration changes take effect on restart. `config.yml` is git-ignored — commit only `config.example.yml`.
</Info>

## Resolution order

Config is merged per key from layered sources, higher overriding lower:

| Priority | Source | Notes |
| -------- | ------ | ----- |
| 1 (highest) | **Environment variables** | Any key can be overridden by an env var of the same name. |
| 2 | **`config.yml`** | Your local file (git-ignored). |
| 3 | **Remote config** | Pulled only when `CONFIG_SERVER`, `CONFIG_TOKEN`, and `ENV` are all set. |
| 4 (lowest) | **`config.example.yml`** | The committed defaults template. |

### Encrypted secrets

Any string value beginning with `gAAAA` is treated as a Fernet token and decrypted on load when `CONFIG_ENCRYPTION_KEY` is set (for both `config.yml` and remote config). The loader treats `CONFIG_ENCRYPTION_KEY` as a raw secret, trims it, takes at most 32 bytes, pads shorter values, and derives the Fernet key. Encryption is the operator's job: nothing rewrites plaintext at rest. If you generate ciphertext outside the server, reproduce `ConfigStore::derive_fernet_key` exactly. Values whose **key name** contains `_KEY`, `_PASSWORD`, `_SECRET`, `_TOKEN` (and similar) are also masked in the config summary.

```bash
# Generate an exact 32-character raw secret
openssl rand -hex 16

# Provide it via the environment (not stored in config.yml)
export CONFIG_ENCRYPTION_KEY="<your-key>"
```

<Note>
`PG_ENCRYPTION_KEY` is the key that encrypts data columns, so it must stay readable to bootstrap the database. Keep it in the environment or a secrets manager in production rather than in `config.yml`.
</Note>

## HTTP server

```yaml config.yml
HTTP_HOST: '0.0.0.0'      # listen on all interfaces
HTTP_PORT: 8080           # default port
# HTTP_URI_PREFIX: ''     # mount the whole app under a sub-path, e.g. '/mirobody'
HTTP_ROOT: 'res/htdoc'    # static web client served for non-API GETs

# CORS (uncomment to enable for browser clients)
# HTTP_HEADERS:
#   Access-Control-Allow-Origin: '*'
#   Access-Control-Allow-Methods: 'GET, POST, PUT, DELETE, OPTIONS'
#   Access-Control-Allow-Headers: 'Authorization, Content-Type, X-Timezone'
#   Access-Control-Max-Age: '86400'
```

<Tip>
Auth is a bearer token in the `Authorization` header (not cookies), so `Access-Control-Allow-Origin: '*'` is fine for local use. Do not combine `'*'` with `Access-Control-Allow-Credentials: 'true'` — browsers reject that; for credentials, use a fixed origin. In production, restrict the origin to your domain.
</Tip>

## Logging

```yaml config.yml
LOG_LEVEL: 'INFO'     # debug | info | warning | error | critical (case-insensitive)
```

Logs always go to **stderr** — there is no log-file option; capture them with your process manager (`docker logs`, journald, a systemd unit, …).

## LLM providers

Set **at least one** provider key to enable chat. The agent picks a client by **provider** name at request time; the default provider fallback is `gemini-2.5-flash`. OpenAI and Gemini run the built-in MCP tools **locally in-process**.

### Google Gemini

```yaml config.yml
GOOGLE_API_KEY: 'AIza...'    # AI Studio key (also read as GEMINI_API_KEY)
```

To run Gemini through your own Google Cloud project and region, use **Vertex AI** instead of AI Studio:

```yaml config.yml
GOOGLE_GENAI_USE_VERTEXAI: '1'
VERTEX_ACCESS_TOKEN: ''      # OAuth access token, e.g. `gcloud auth print-access-token` (rotate ~hourly)
GCP_PROJECT: ''              # Google Cloud project id (also read as GOOGLE_CLOUD_PROJECT)
VERTEX_LOCATION: 'us-central1'
# VERTEX_BASE_URL: ''        # optional host override (sovereign clouds / regions)
```

### OpenAI

```yaml config.yml
OPENAI_API_KEY: 'sk-...'
# OPENAI_BASE_URL: ''        # point at any OpenAI-compatible endpoint (OpenRouter, DeepSeek, Groq, …)
```

To run GPT through your own Azure resource and region, use **Azure OpenAI**. The client switches to Azure mode when `AZURE_OPENAI_ENDPOINT` is set:

```yaml config.yml
AZURE_OPENAI_ENDPOINT: 'https://my-resource.openai.azure.com'
AZURE_OPENAI_DEPLOYMENT: 'gpt-4o-mini'
AZURE_OPENAI_API_VERSION: '2024-10-21'
AZURE_OPENAI_API_KEY: ''     # or AZURE_OPENAI_KEY
```

### On-device Gemma (optional)

The native apps can run **Gemma 4 E2B** fully on-device (LiteRT-LM on Android/iOS, llama.cpp on Electron) — no key, no network. For a self-hosted Gemma runtime (e.g. Ollama), point the standalone server at it:

```yaml config.yml
# GEMMA_CHAT_BASE_URL: 'http://localhost:11434/v1'   # falls back to OLLAMA_BASE_URL
# GEMMA_CHAT_MODEL: 'gemma-4-e2b'                     # the tag your runtime serves
# GEMMA_CHAT_API_KEY: ''                              # usually empty for a local server
```

### Embeddings

```yaml config.yml
EMBEDDING_PROVIDER: 'gemini'   # embeddings provider (uses the Gemini/Google key above)
```

## Database

Exactly one SQL backend is compiled into the binary (chosen at build time via `MIROBODY_DATABASE_BACKEND` — see [Installation](/en/installation#database-backends)). These keys configure the **PostgreSQL** backend, the desktop default:

```yaml config.yml
PG_HOST: '127.0.0.1'
PG_PORT: 5432
PG_USER: 'postgres'
PG_DBNAME: 'mirobody'
PG_SCHEMA: 'public'
PG_PASSWORD: 'REPLACE_THIS_VALUE_IN_PRODUCTION'
PG_ENCRYPTION_KEY: 'REPLACE_THIS_VALUE_IN_PRODUCTION'   # encrypts sensitive data columns
```

The SQLite backend (mobile / self-contained builds) needs no connection settings. MySQL, DuckDB, and ClickHouse have their own analogous key blocks.

## Cache

The cache layer is **optional**. Leave `REDIS_HOST` empty and the server uses an in-process in-memory KV store — fine for single-instance runs. Set `REDIS_HOST` to share cache state across multiple instances:

```yaml config.yml
# REDIS_HOST: ''
# REDIS_PORT: 6379
# REDIS_PASSWORD: ''
```

## File storage

Uploads and generated assets (e.g. charts) go to a storage backend, selected at runtime. The default is the **local filesystem**; the others (S3 / S3-compatible, Alibaba OSS, Azure Blob) are picked by filling their block.

<Tabs>
  <Tab title="Local (default)">
    ```yaml config.yml
    LOCAL_STORAGE_DIR: '_local/storage'          # where objects are written
    LOCAL_STORAGE_URL_PREFIX: '/files'           # URL path this server serves objects at
    LOCAL_STORAGE_SECRET: 'REPLACE_THIS_VALUE_IN_PRODUCTION'  # HMAC key for presigned URLs (>=16 chars when URL prefix set)
    # LOCAL_STORAGE_PREFIX: ''                   # object-key prefix (empty => 'mirobody')
    # LOCAL_STORAGE_BASE_URL: ''                 # public prefix / CDN base
    ```
  </Tab>

  <Tab title="S3 / S3-compatible">
    ```yaml config.yml
    S3_KEY: ''         # AWS access key id
    S3_TOKEN: ''       # AWS secret access key
    S3_REGION: ''      # e.g. us-east-1
    S3_BUCKET: ''
    # S3_ENDPOINT: ''  # host override for S3-compatible stores (MinIO, R2, GCS interop); empty => AWS
    # S3_PREFIX: ''    # key prefix (empty => 'mirobody')
    ```
  </Tab>

  <Tab title="Alibaba OSS">
    ```yaml config.yml
    ALI_OSS_ACCESS_KEY: ''
    ALI_OSS_SECRET_KEY: ''
    ALI_OSS_ENDPOINT: 'oss-cn-hangzhou.aliyuncs.com'
    ALI_OSS_BUCKET_NAME: ''
    # ALI_OSS_PREFIX: ''
    ```
  </Tab>

  <Tab title="Azure Blob">
    ```yaml config.yml
    AZURE_BLOB_ACCOUNT: ''
    AZURE_BLOB_KEY: ''                          # account key (base64)
    AZURE_BLOB_CONTAINER: ''
    # AZURE_BLOB_ENDPOINT_SUFFIX: 'core.windows.net'   # override for sovereign clouds
    ```
  </Tab>
</Tabs>

<Info>
Optional at-rest encryption for uploads: set `FILE_ENCRYPTION_KEY` (a rotatable list — the last key encrypts, all keys decrypt) plus a stable `FILE_KEY_SEED`. See the comments in `config.example.yml`.
</Info>

## Long-term memory

The `remember` / `recall_memory` tools store durable facts. Every backend compiles in; pick one at runtime:

```yaml config.yml
MEMORY_PROVIDER: 'local'   # local (default) | mem0 | zep | everos
```

`local` keeps facts plus 1024-dim embeddings in the app database, ranked by in-process cosine similarity — no extra services. The others talk to a remote memory service over HTTP.

## Vendor OAuth tokens

Per-user vendor access and refresh tokens are stored only when an at-rest encryption key is available. `VENDOR_TOKEN_ENCRYPTION_KEY` accepts a list of 44-character Fernet keys; the last encrypts new tokens and all listed keys can decrypt during rotation. When unset, it falls back to `FILE_ENCRYPTION_KEY`.

```yaml config.yml
VENDOR_TOKEN_ENCRYPTION_KEY:
  - '<44-character Fernet key>'
VENDOR_REDIRECT_URI: 'https://api.example.com/vendors/callback'
```

Generate a compatible key with `./build/fernet genkey`. If neither vendor-token nor file encryption is configured, an OAuth callback can prove ownership but the server deliberately does not store bearer tokens in plaintext; later fetches must rely on configured shared credentials. Multi-user or multi-instance OAuth deployments therefore need one of these keys.

## Authentication

### Email one-time-code

The web client's default sign-in. For local testing, preset codes:

```yaml config.yml
EMAIL_PREDEFINE_CODES:
  demo1@mirobody.ai: '777777'
  demo2@mirobody.ai: '777777'
  demo3@mirobody.ai: '777777'
```

<Warning>
Never ship predefined codes in production.
</Warning>

### JWT

```yaml config.yml
JWT_KEY: 'REPLACE_THIS_VALUE_IN_PRODUCTION'    # HS256 secret (256 bits): `openssl rand -hex 32`
JWT_SALT: 'REPLACE_THIS_VALUE_IN_PRODUCTION'
# JWT_PRIVATE_KEY: |                            # set for RS256 — takes precedence over JWT_KEY, publishes a JWKS
#   -----BEGIN PRIVATE KEY-----
#   ...
```

Set exactly one signing mode: `JWT_KEY` for HS256, or `JWT_PRIVATE_KEY` for RS256 (which enables third-party token verification via the published JWKS). The `cli/jwt_keygen` tool generates an RSA keypair.

### Social sign-in (optional)

Google, Apple, and X sign-in go through **Firebase**; WeChat and GitHub are direct; **Tanka QR-code login** is on by default.

```yaml config.yml
FIREBASE_PROJECT_ID: ''          # verifies Google/Apple/X ID tokens
# FIREBASE_WEB_API_KEY: ''       # web app browser apiKey
# FIREBASE_MESSAGING_SENDER_ID: ''
# APPLE_CLIENT_ID: ''
# WECHAT_APPID: ''
# WECHAT_SECRET: ''
# GITHUB_CLIENT_ID: ''
# GITHUB_CLIENT_SECRET: ''
# TANKA_LOGIN_ENABLED: true      # QR login; set false to hide the button
```

## OAuth 2.0 authorization server

Mirobody embeds an OAuth 2.0 authorization server (authorization-code + PKCE) so MCP clients can obtain tokens through the browser instead of pasting a bearer token. It works with the JWT keys above; all `OAUTH_*` keys have working defaults.

```yaml config.yml
# JWT_ISS: ''          # iss/aud/client_id/scope claims minted into self-signed tokens
# JWT_AUD: ''
# OAUTH_RESOURCE_PATH: '/mcp'   # the resource these tokens protect
```

Discovery documents are served at `/.well-known/oauth-authorization-server`, `/.well-known/openid-configuration`, and `/.well-known/jwks.json`.

## Remote MCP access

Remote MCP clients need no extra configuration on this server: point them at your public HTTPS base URL plus `/mcp`, and the MCP/OAuth services derive their advertised URLs from each request's `Host` header. For local development without a public domain, tunnel it:

<Steps>
  <Step title="Install and run ngrok">
    ```bash
    ngrok http 8080
    ```
  </Step>
  <Step title="Point the client at the tunnel">
    Give the client `https://abc123.ngrok-free.app/mcp` as the server URL.
  </Step>
</Steps>

## Care circles

Opt-in sharing on top of the single-user core. Growth is bounded by caps (`0` disables either):

```yaml config.yml
CIRCLE_MAX_PER_USER: 5    # max circles a user may own
CIRCLE_MAX_MEMBERS: 5     # max members per circle (pending invites count)
```

Sharing is off by default; the AI assistant is **read-only** over another member's data. See the [architecture overview](/en/concepts/architecture).

## Chat rate limiting

Per-user cap on agent / realtime turns:

```yaml config.yml
CHAT_RATE_MAX: 5           # max turns per user per window (0 = unlimited)
CHAT_RATE_WINDOW_SEC: 60   # window length in seconds
```

## Remote config (optional)

Pull config from a service instead of a local file — active only when all three are set:

```yaml config.yml
# via environment:
# CONFIG_SERVER=https://config.example.com
# CONFIG_TOKEN=<bearer sent as X-Config-Token>
# ENV=prod                  # selects the remote environment
```

## Configuration best practices

<AccordionGroup>
  <Accordion title="Version control" icon="git">
    - ✅ Commit `config.example.yml`
    - ❌ Never commit `config.yml` with real credentials (it is git-ignored by default)
    - ✅ Provide `CONFIG_ENCRYPTION_KEY` via the environment, not the file
  </Accordion>

  <Accordion title="Security" icon="shield">
    - Replace every `REPLACE_THIS_VALUE_IN_PRODUCTION` placeholder with a strong, random value
    - Use RS256 (`JWT_PRIVATE_KEY`) when other services must verify your tokens
    - Restrict CORS origins in production
    - Inventory every outbound data flow and obtain any agreements your workload requires; selecting Vertex AI or Azure OpenAI does not by itself establish compliance
  </Accordion>

  <Accordion title="Validate YAML" icon="circle-check">
    ```bash
    python3 -c "import yaml; yaml.safe_load(open('config.yml'))"
    ```
    No output means the syntax is valid. The server also reports config errors on startup rather than silently degrading.
  </Accordion>
</AccordionGroup>
