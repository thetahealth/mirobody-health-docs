Mirobody is a **C++11** codebase built with **CMake** and **Ninja** (vcpkg on Windows). Developing on it means installing the native toolchain, building the `mirobody` target, and running the Catch2 test suite.

## Prerequisites

## Setup

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/thetahealth/mirobody.git
    cd mirobody
    ```
  </Step>

  <Step title="Install the native dependencies">
    vcpkg is Windows-only; Linux and macOS use the system package manager.

    <Tabs>
      <Tab title="Debian / Ubuntu / WSL">
        ```bash
        sudo apt install build-essential cmake ninja-build pkg-config \
                         libwebsockets-dev libcurl4-openssl-dev libssl-dev \
                         rapidjson-dev libyaml-cpp-dev libhiredis-dev libpq-dev \
                         libjpeg-dev libpng-dev libtiff-dev libwebp-dev

        # Catch2 v3 for the test suite
        sudo apt install catch2
        ```
      </Tab>

      <Tab title="macOS (Homebrew)">
        ```bash
        brew install cmake ninja pkg-config libwebsockets curl openssl@3 \
                     rapidjson yaml-cpp hiredis libpq \
                     jpeg-turbo libpng libtiff webp catch2
        ```
      </Tab>

      <Tab title="Fedora / RHEL">
        ```bash
        sudo dnf install gcc-c++ cmake ninja-build pkgconf-pkg-config \
                         libwebsockets-devel libcurl-devel openssl-devel \
                         rapidjson-devel yaml-cpp-devel hiredis-devel libpq-devel \
                         libjpeg-turbo-devel libpng-devel libtiff-devel libwebp-devel \
                         catch2-devel
        ```
      </Tab>
    </Tabs>

    <Info>
    On Windows, install Visual Studio (Desktop development with C++ + C++ CMake tools). vcpkg pulls every dependency — including Catch2 — from `vcpkg.json` automatically. See [Installation](/en/installation).
    </Info>
  </Step>

  <Step title="Build">
    The `build.sh` wrapper configures (Release, PostgreSQL backend, tests ON by default) and builds:

    ```bash
    ./build.sh              # -> build/mirobody  and  build/tests/mirobody_tests
    ```

    Pick a different SQL backend with a token (`sqlite`, `legacy`, `mysql`, `duckdb`, `ck`); each backend gets its own `build[-<backend>]` dir so several can coexist:

    ```bash
    ./build.sh sqlite       # -> build-sqlite/
    ./build.sh clean        # wipe and reconfigure
    ```

    Or drive CMake directly for finer control (a Debug build, extra flags):

    ```bash
    cmake -S . -B build -G Ninja \
        -DCMAKE_BUILD_TYPE=Debug \
        -DMIROBODY_DATABASE_BACKEND=POSTGRESQL
    cmake --build build
    ```
  </Step>

  <Step title="Configure and run">
    ```bash
    cp config.example.yml config.yml
    ```

    Set an LLM key (and, for the default backend, your `PG_*` connection) in `config.yml`, then run:

    ```bash
    ./build/mirobody --config config.yml
    ```

    The server starts on `http://localhost:8080`. See [Configuration](/en/configuration) for the full key list.

    <Tip>
    For a self-contained dev loop with no external database, build the SQLite backend (`./build.sh sqlite`) and run `./build-sqlite/mirobody`.
    </Tip>
  </Step>
</Steps>

## Useful CMake options

| Option | Default | Purpose |
| ------ | ------- | ------- |
| `MIROBODY_DATABASE_BACKEND` | `POSTGRESQL` | Which SQL backend to link (`SQLITE`, `MYSQL`, `DUCKDB`, `CLICKHOUSE`, `POSTGRESQL_LEGACY`) |
| `MIROBODY_BUILD_TESTS` | `ON` | Build the Catch2 test suite |
| `MIROBODY_BUILD_TOOLS` | `ON` | Build the `cli/` debug binaries |
| `MIROBODY_ENABLE_PDF` | `OFF` | PDF text extraction via PDFium |
| `MIROBODY_ENABLE_OCR` | `OFF` | OCR via Tesseract (implies PDF) |
| `MIROBODY_ENABLE_XLS` | `OFF` | Legacy `.xls` via libxls |
| `MIROBODY_BUILD_SHARED` / `_PYTHON` / `_JNI` | varies | Build the shared library / Python wheel / JNI shim instead of (or alongside) the binary |

## Running tests

Tests are a Catch2 executable (`mirobody_tests`) registered with CTest via `catch_discover_tests`.

```bash
# Build (tests are on by default)
./build.sh

# Run via CTest
ctest --test-dir build --output-on-failure

# Or run the binary directly (Catch2 filters / tags)
./build/tests/mirobody_tests
./build/tests/mirobody_tests "[fhir]"      # only the fhir tests
./build/tests/mirobody_tests --list-tests
```

Test sources live under `tests/` (subdirectories like `chat/`, `fhir/`, `circle/`, `jwt/`, `cache/`, plus an `e2e/` suite).

## Debug CLIs

The desktop build produces standalone per-subsystem CLIs under `cli/` (toggle with `-DMIROBODY_BUILD_TOOLS=OFF`) that bypass the HTTP/WS server — handy for poking at one piece in isolation:

- **LLM clients** — `openai_chat`, `openai_responses`, `openai_realtime`, `gemini`, `gemini_live`
- **Storage** — `aws_s3`, `aliyun_oss`, `azure_blob`
- **Transcoders** — `image`, `document`, `file_parser`
- **FHIR** — `fhir normalize "<5.6 mg/dL"`, and `fhir token <user_id> --config <yaml>` to mint a bearer JWT for the FHIR routes
- **Auth** — `jwt_keygen` (RSA keypair for RS256 / JWKS), `google_jwt`, `firebase_jwt`
- **Other** — `mcp`, `agent`, `vendor`

See `cli/README.md` in the repo for the per-binary flags and credentials.

## Adding an agent or MCP tool

Both self-register at **compile time** — there is no runtime plugin directory:

- **Agent** — the single agent is `res/agents/baseline.cpp`. Every `.cpp` under `res/agents/` is globbed into the build.
- **MCP tool** — drop a `.cpp` in `res/mcp_tools/` with a `MIROBODY_REGISTER_TOOL(...)` line declaring its parameter table, then rebuild. See `res/mcp_tools/echo.cpp` for the smallest example and [Adding Tools](/en/tools/adding-tools).

```bash
# after adding a file under res/, just rebuild
./build.sh
```

## Project layout

```
mirobody/
├── src/            # C++ core (server, chat, llm, mcp, fhir, health/vendor, database, storage, …)
│   ├── mirobody.h  # public C ABI
│   └── main.cpp    # standalone server entry point
├── res/            # compiled/served resources: agents/, mcp_tools/, mcp_resources/, sql/, htdoc/
├── tests/          # Catch2 unit tests (build/tests/mirobody_tests)
├── cli/            # standalone per-subsystem debug binaries
├── htdoc/          # web client source (webpack -> res/htdoc)
├── bindings/       # Java / Go / C# / Node / Rust FFI examples
├── android/  ios/  electron/  qt/  miniapp/   # host apps / clients
├── CMakeLists.txt  build.sh  build.cmd
└── config.example.yml
```

See the full tree and the other build artifacts in [Installation → Repository layout](/en/installation#repository-layout).
