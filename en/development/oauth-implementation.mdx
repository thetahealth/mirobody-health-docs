## Overview

Most health-data sources authorize with **OAuth 2.0**. In Mirobody this is handled inside each C++ `vendor::Vendor` via five contract operations ÔÇö `authorize_url`, `exchange_code`, `refresh`, `handle_webhook`, `revoke` ÔÇö plus, for browser-driven SMART-on-FHIR EHRs, a separate stateful connect service. There is no `OAuth1Session` / `aiohttp` provider code any more.

<Info>
Two OAuth roles exist in the codebase, don't confuse them:
- **`src/oauth/`** ÔÇö Mirobody as an OAuth 2.0 **authorization server** (it issues tokens to API clients; OIDC discovery + PKCE).
- **Vendor clients** ÔÇö Mirobody as an OAuth **client** *of* a health source (Fitbit, an EHR, ÔÇª). This page is about the client side.
</Info>

## OAuth 2.0 ÔÇö the `authorize_url` step

`authorize_url` builds the consent URL you redirect the user to. It's **pure string construction** ÔÇö no network call. Fitbit (`device/fitbit.cpp`) is the reference: a confidential server client authenticates with its secret at the token step, so **PKCE is not used** here.

```cpp
std::string authorize_url(const std::string& redirect_uri, const std::string& state,
                          const std::string& /*user_id*/, const std::string& /*provider*/) override {
    if (config().client_id.empty()) {
        throw VendorError(info_.id + ": authorize_url requires an OAuth client_id ÔÇö set "
                          "MIROBODY_VENDOR_FITBIT_CLIENT_ID");
    }
    std::string url = "https://www.fitbit.com/oauth2/authorize"
                      "?response_type=code&client_id=" + url_encode(config().client_id) +
                      "&scope=" + url_encode("activity heartrate sleep weight profile");
    if (!redirect_uri.empty()) url += "&redirect_uri=" + url_encode(redirect_uri);
    if (!state.empty())        url += "&state=" + url_encode(state);
    return url;
}
```

- `state` is your correlation / CSRF value, echoed back on the callback.
- The **token exchange and refresh are implemented in the client**: `exchange_code(code, redirect_uri)` and `refresh(refresh_token)` POST to the vendor's token endpoint through the shared helper `oauth2_token_request(...)` in [`src/health/vendor/oauth2.cpp`](https://github.com/thetahealth/mirobody/blob/main-v2/src/health/vendor/oauth2.cpp), which sends the form-encoded RFC 6749 request (client credentials in the body or via HTTP Basic) and parses the `{access_token, refresh_token?, expires_in?}` JSON into a `TokenSet`. Fitbit, Dexcom, Oura, Whoop, and Polar all go through it; Withings speaks its own `{status, body}` envelope and implements the pair itself. **Polar's `refresh` is an honest stub** ÔÇö its access tokens never expire, so there is no refresh grant. The resulting access token is what rides as `config.api_key` (`<ID>_API_KEY`) on `fetch`.

## Revocation

`revoke` drops the grant. Fitbit POSTs the token to `/oauth2/revoke` authenticated as the confidential client (HTTP Basic `client_id:client_secret`):

```cpp
req.url     = base_url() + "/oauth2/revoke";
req.body    = "token=" + url_encode(config().api_key);
req.headers = { "Authorization: Basic " +
                storage::base64_encode(config().client_id + ":" + config().client_secret) };
```

Garmin's `revoke` instead calls its documented account-disconnect hook (`DELETE /wellness-api/rest/user/registration`, Bearer). Withings has no public token-revoke endpoint, so its `revoke` is an honest stub.

## Verifying webhooks

When a source signs its webhook, verify it before trusting the body. Fitbit signs with `X-Fitbit-Signature = base64(HMAC-SHA1(raw_body))` keyed by `client_secret + "&"`; the client recomputes and compares in constant time:

```cpp
const std::string key = config().client_secret + "&";
std::array<unsigned char, 20> mac = storage::hmac_sha1(key, body);
if (!ct_equal(storage::base64_encode(mac.data(), mac.size()), sig)) {
    throw VendorError(info_.id + ": webhook signature verification failed (X-Fitbit-Signature)");
}
```

If the signature scheme **isn't public** (as with Garmin's unsigned push), `handle_webhook` stays a stub ÔÇö Mirobody won't fabricate a verification scheme.

## PKCE and stateful flows (SMART on FHIR, Garmin)

Some flows need **PKCE**, where a `code_verifier` minted at authorize-time must be persisted and replayed at the token exchange. A bare `authorize_url` has nowhere to stash the verifier, so PKCE consent lives in a **stateful connect layer**, not in the vendor's `authorize_url`. That's why Garmin's `authorize_url` is a stub.

The shipped example is the **SMART-on-FHIR EHR connect service** (`src/health/ehr_connect.*`), where Mirobody is an OAuth client of a hospital EHR:

<Steps>
  <Step title="Authorize">
    `POST /health/ehr/authorize {fhir_base_url}` runs **SMART discovery** (`{base}/.well-known/smart-configuration`), generates the PKCE verifier + `state`, **caches** them, and returns the tenant's `authorize_url` for the browser.
  </Step>
  <Step title="Callback">
    `GET /health/ehr/callback?code=&state=` ÔÇö no bearer; the user is recovered from the single-use `state`. The cached verifier is replayed to exchange the `code` at the tenant's token endpoint, and the access token is cached per user.
  </Step>
  <Step title="Sync">
    `POST /health/ehr/sync` uses the cached token to fetch `Observation`s via the `ehr` client and persists them.
  </Step>
</Steps>

Configured by the `SMART_FHIR_*` keys (client id / redirect uri / scope). PKCE generation itself is unit-tested (`tests/oauth/pkce_test.cpp`).

## Security notes

<AccordionGroup>
  <Accordion title="Token storage" icon="shield">
    Access tokens live in `VendorConfig.api_key`, sourced from the `<ID>_*` config keys. Store secrets pre-encrypted as Fernet (`gAAAAÔÇª`) values ÔÇö they are transparently decrypted on load when `CONFIG_ENCRYPTION_KEY` is set ÔÇö and per-user vendor tokens from the bind flow are persisted only encrypted (`VENDOR_TOKEN_ENCRYPTION_KEY`). Never log tokens.
  </Accordion>
  <Accordion title="State / CSRF" icon="fingerprint">
    Pass a random `state` to `authorize_url` and verify it on callback. The EHR connect service uses a single-use cached `state` that also carries the user binding.
  </Accordion>
  <Accordion title="Constant-time comparison" icon="clock">
    Compare webhook signatures with a constant-time compare (`ct_equal`), never `==`.
  </Accordion>
  <Accordion title="Don't guess contracts" icon="triangle-exclamation">
    No public signature scheme ÔåÆ stubbed webhook. No documented revoke ÔåÆ stubbed revoke. Honesty over fabrication.
  </Accordion>
</AccordionGroup>
